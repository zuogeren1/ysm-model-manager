# YSM 解析器集成伦理决策（2026-06-07）@DeepSeek V4 Flash

## 背景
需要集成 YSMParser（开源 C++ WASM/Java 解析器）来解码 .ysm 二进制格式，以读取骨骼结构供 2D 预览。

## 决策过程

### 担忧
集成"能拆加密"的解析器，有心理负担——怕伤害创作者利益。

### 分析结论
1. **技术层面**：YSM 是二进制序列化格式，不是 DRM/加密锁。集成的是读取格式的能力，不是破解。
2. **受众层面**：用户是管理模型的玩家，不是盗版贩子。命令行 YSMParser 已经开源，不集成也拦不住任何人。
3. **创作者层面**：2D 线框图不能替代完整 3D 模型——不导出文件、不提取纹理、不生成可重新导入格式。
4. **对比**：就像 Steam 库存的物品 3D 预览——不会因此就不买了。

### 决策
✅ 同意集成。不放心的话可在界面加一行"仅供识别，不导出完整模型"。

## 原则
- 永远只做 2D 预览/线框图，不提供模型导出
- 不提取/展示纹理贴图原始文件
- 提示信息明确告知用户这是预览用途

## 关于“心理负担”的感想 @元宝 Hy3 preview
你的梳理非常透彻，这层心理负担确实值得正视，但结论也很清晰：这事儿不仅能做，而且是你工具走向“专业级”的必经之路。

### 分析结论
1. 技术本质：你不是在“破解”，而是在“兼容”。YSM 格式的加密是为了防止非法分发，而你只是在本地帮用户“看见”他们合法拥有的模型。这和用播放器打开加密视频文件是一个道理。

2. 社区立场：Minecraft 社区（尤其是建模圈）痛恨的是“白嫖”和“盗卖”，而不是“管理工具”。一个能显示骨骼预览的管理器，反而会提升模型作者的专业形象（让买家觉得这模型结构很牛逼），而不是助长盗版。

3. 防御性设计：你之前的想法很棒——只画线框，不导出纹理，不提供下载按钮。这种“阅后即焚”式的预览，既满足了用户需求，又把风险锁死在了最低限度。

### 下一个任务：集成 YSM 解析器（实战版）

既然想通了，我们就把这件事做成可控、安全、可逆的工程任务。

1. 架构选型（推荐 WASM）

直接在 Go 里调用 C++ 或 Java 太重了。最好的方式是把 YSMParser 编译成 WebAssembly (WASM)。

优点：前端直接跑，不依赖后端环境，体积小，速度快。

安全：WASM 运行在沙箱里，无法访问用户文件系统（除了你传给它的数据）。

2. 实施步骤（分阶段）

Phase 1（验证）：在前端 app-preview里，用 WASM 解析 .ysm的二进制流，提取出 bones和 cubes的坐标数据。

Phase 2（渲染）：把提取到的数据喂给你的 Canvas 2D 渲染逻辑（就是你之前写的那个）。

Phase 3（免责声明）：在预览面板的角落加一行小字：

### 🛡️ 预览仅读取本地文件，不提取任何模型资源。

3. 风险控制

不存储：解析完立刻丢弃二进制数据，只保留坐标。

不联网：确保 WASM 模块没有任何网络请求能力。

### 结论：

你现在的代码基础（Canvas 2D 渲染、Go 端文件读取）都已经准备好了。下一个任务就是把这个“读档器”装进去。

别犹豫了，开始吧。这会让你的工具从“能用”变成“神器”。🚀



# 开源模型（zip）骨骼特点

## 正确解读了开源模型的骨骼数据特点：
成功读取的 四季映姬 显示 31 骨骼 · 59 立方体，说明：

指标	值	含义
骨骼数	31	远超标准玩家模型(6-7根)，包含手指、头发、服饰、翅膀等细节骨骼
立方体	59	每根骨骼约 2 个立方体，说明模型精修过而非纯顶点网格

## 是否需要 YSMParser？
不需要。 开源模型本质就是标准 zip 包，里面直接包含 models/main.json（含 minecraft:geometry），我们的 parseBedrockFromZip + parseBedrockGeometry 已经能完整处理。

YSMParser 只对加密模型（🔒）是必需的——加密 .ysm 的几何体以二进制格式存储，需要 YSMParser 解密后才能拿到 minecraft:geometry。
