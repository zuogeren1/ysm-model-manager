# YSM 解析器集成伦理决策（2026-06-07）@DeepSeek V4 Flash

## 背景
需要集成 YSMParser（开源 C++ WASM/Java 解析器）来解码 .ysm 二进制格式，以读取骨骼结构供 2D 预览。

## 决策过程

### 担忧
集成"能拆加密"的解析器，有心理负担——怕伤害创作者利益。

### 分析结论
1. **技术层面**：YSM 是二进制序列化格式，不是 DRM/加密锁。集成的是读取格式的能力，不是破解。
2. **受众层面**：用户是管理模型的玩家，不是盗版贩子。命令行 YSMParser 已经开源，不集成也拦不住任何人。
3. **创作者层面**：2D 线框图不能替代完整 3D 模型——不导出文件、不提取纹理、不生成可重新导入格式。
4. **对比**：就像 Steam 库存的物品 3D 预览——不会因此就不买了。

### 决策
✅ 同意集成。不放心的话可在界面加一行"仅供识别，不导出完整模型"。

## 原则
- 永远只做 2D 预览/线框图，不提供模型导出
- 不提取/展示纹理贴图原始文件
- 提示信息明确告知用户这是预览用途

下一个任务：集成 YSM 解析器（实战版）

既然想通了，我们就把这件事做成可控、安全、可逆的工程任务。

1. 架构选型（推荐 WASM）

直接在 Go 里调用 C++ 或 Java 太重了。最好的方式是把 YSMParser 编译成 WebAssembly (WASM)。

优点：前端直接跑，不依赖后端环境，体积小，速度快。

安全：WASM 运行在沙箱里，无法访问用户文件系统（除了你传给它的数据）。

2. 实施步骤（分阶段）

Phase 1（验证）：在前端 app-preview里，用 WASM 解析 .ysm的二进制流，提取出 bones和 cubes的坐标数据。

Phase 2（渲染）：把提取到的数据喂给你的 Canvas 2D 渲染逻辑（就是你之前写的那个）。

3. 风险控制

不存储：解析完立刻丢弃二进制数据，只保留坐标。

不联网：确保 WASM 模块没有任何网络请求能力。
