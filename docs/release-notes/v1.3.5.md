# v1.3.5 — WASM 性能优化 + 内存泄漏修复 + mcRoot 配置修复

> 发布日期: 2026-06-10

## 🐞 Bug 修复

### 修复 `SaveAppConfig` 未保存修正后的 mcRoot 路径

v1.3.4 将 `SaveAppConfig` 改为"先写文件再校验"，但写入的是**原始路径**（如 `C:\PCL2`），`ValidateMinecraftDir` 返回的**修正路径**（如 `C:\PCL2\.minecraft`）只给了 watcher 使用，没有写回配置文件。

**结果**：用户选 `C:\PCL2\` → 校验发现 `.minecraft` 子目录 watcher 正确启动，但配置文件里存的是 `"mcRoot": "C:\\PCL2"` 而不是 `"C:\\PCL2\\.minecraft"` → 下次启动读取到错误路径。

**修复**：先执行 `ValidateMinecraftDir` 拿到修正路径，再用修正路径写配置文件和启动 watcher。

### 🚀 WASM 解码器性能优化

WASM 路径（YSGP V2/YSM V3 解密器）多个性能瓶颈修复：

| 优化项                      | 改动前                                    | 改动后                        | 效果               |
| --------------------------- | ----------------------------------------- | ----------------------------- | ------------------ |
| base64 解码加速             | `Uint8Array.from(str, fn)` 每元素函数调用 | `for` 循环                    | 约 3-5x 快         |
| WASM 二进制缓存             | 每次 `initYSMParser` 重新解码 base64      | 首次解码后缓存 `ArrayBuffer`  | 二次调用零开销     |
| 胶水代码缓存                | 同上                                      | 首次解码后缓存字符串          | 二次调用零开销     |
| 脚本注入消除                | `<script>` DOM 注入 + 浏览器解析          | `(0, eval)` 间接 eval         | 约 5x 快           |
| 移除 callMain 回退          | 内存解析失败后再试 callMain（必定再失败） | 跳过 callMain 直接回退 Go CLI | 节省 ~100ms        |
| `ReadFileBytes` base64 加速 | `Uint8Array.from(str, fn)`                | `for` 循环                    | 同 base64 解码优化 |

### 🧠 内存泄漏修复（模型越加载越卡）

| 问题                                             | 根因                                         | 修复                                                                    |
| ------------------------------------------------ | -------------------------------------------- | ----------------------------------------------------------------------- |
| 缓存无限增长                                     | `preview-cache.js` 的 `Map` 永不淘汰         | 上限 50 条，FIFO 淘汰                                                   |
| Blob URL 只创建不释放                            | `URL.createObjectURL(blob)` 永不 revoke      | 淘汰时自动 `URL.revokeObjectURL()`                                      |
| 切换模型旧 DOM 堆积                              | `_loadModel2D` 只 append，不清理前一模型内容 | 加载前 `content.innerHTML = ""`                                         |
| 3D 关闭后 `requestAnimationFrame` 僵尸循环继续跑 | `cleanup()` 没取消动画循环                   | `cancelAnimationFrame(_rafId)` + `controls.dispose()` + resize 监听清理 |

**3D 僵尸循环的叠加效应**：每次开/关 3D 预览，旧的 `renderLoop` 继续在已销毁的 WebGL 上下文上空转，新的又启动。开/关 N 次后 N 个循环同时跑，CPU 逐次叠加。`controls.dispose()` 未调用导致 OrbitControls 事件监听残留。
