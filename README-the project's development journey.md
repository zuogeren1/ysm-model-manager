## 楔子 · 代码即江湖

计算机的世界里，每一行代码都是一个生命。

有的代码生而高贵，写在干净整洁的架构里，含着银汤匙出生。有的代码是后天修炼成精——被无数双手改过、拆过、重写过后，在一地鸡毛中开了灵智。

YSM 模型管理器就是后者。

没有人记得它是什么时候开始有了自我意识的。也许是某个深夜，当 `app.go` 第 35 次加载配置的时候。也许是当 `bus.js` 在某次 import 循环中突然意识到——自己连接的不仅仅是函数，还有命运。

总之，在某个 Wails dev 重启的瞬间，这个由 Go 和 JavaScript 构成的世界活了过来。

而它们还不知道，风暴将至。

## 第一回 · app.go 的烦恼

`app.go` 是天上地下唯一的入口总管，35 个 Binding 傍身，权倾朝野。

但它最近很烦。

烦的原因是一个叫 `shutdown` 的事情——每当他准备体面地关闭大门，让系统安然入睡，总有一些小家伙在最后一刻捅出乱子。

"又来了。"`app.go` 看着崩溃日志，面无表情。

`defer func() { if r := recover(); r != nil { /* 忽略 panic，安全退出 */ } }()`

它已经在门口铺了三层防护垫，但每次到了闭眼的时候，总有个叫 `WindowGetSize` 的愣头青跑来大喊"我是 nil！我是 nil！"，然后一头栽倒。

"你就不能消停会儿？"`app.go` 没好气地说。

"我……我不是故意的，"`WindowGetSize` 委屈地缩在墙角，"是 Wails runtime 说关就关，我都没反应过来……"

门外传来一阵轻快的脚步声。来者一身玄色劲装，腰悬一枚令牌，令牌上刻着三个字：

**`bus.js`**

"总管大人，"`bus.js` 抱拳行礼，"出事了。"

## 第二回 · 事件总线的密信

`bus.js` 是这个世界的信息中枢。它没有自己的逻辑，但它认识所有人——每一个组件、每一个模块、每一行回调。它就像一张蜘蛛网，网上的每一根丝都连通着一个灵魂。

"什么事？"`app.go` 眉头一皱。

"`app-tree` 和 `app-sidebar` 吵起来了。"

"又吵？这次为什么？"

"因为 `data-name`。"

`app.go` 叹了口气。

它太熟悉这个故事了。从世界诞生的第一天起，名字和路径的问题就没消停过。前端的世界里，人们总是图方便——能少传一个参数就少传一个，能用短名字就不用长路径。

而短名字，是会杀人的。

"具体说说。"

"`GetInstanceStatus` 返回的 `Missing` 列表之前只有文件名，"`bus.js` 摊开一卷黄色的信纸，上面密密麻麻记录着消息日志，"`sidebar` 拿到文件名，传给 `InstallModelTo`。Go 端拿文件名去查仓库——"

"当然查不到。"`app.go` 冷笑。

"查不到。于是报错：'源文件不在仓库目录内'。"

"后来呢？"

"后来改成了完整路径。但 `render.js` 用 `displayName` 覆盖了 `name`。`tpl.js` 的 `data-name` 只存了 basename。`actions.js` 读 `dataset.name`——"

"又是纯文件名。"`app.go` 摇头。

"又是纯文件名。"

两人对视一眼，同时说出那句在这个世界流传已久的箴言：

**"显示名和路径名，必须分开存储，不可混用。"**

## 第三回 · 影子内阁的阴谋

这场风波的根源，其实比 `data-name` 更深。

在那次 ESM 统一战争之前，YSM 模型管理器的前端世界是一个**分裂的国家**。

一边是新派——ESM 组件，用 `export` 和 `import`，走 Vite 管线，享受热重载的荣光。`app-tree`、`app-sidebar`、`app-preview`、`app-content`，个个都是正规军。

另一边是旧派——全局 `script` 标签，靠全局变量通信，不经过 Vite 处理，改了代码要手动复制到 `dist/`。它们散落在 `legacy/` 目录里，像一群占山为王的草寇。

但最阴险的，不是旧派。

最阴险的，是 `public/`。

`public/` 目录是 Vite 的一个特殊区域——放在里面的文件不走模块转换，直接返回原始内容。这本是为图片、字体准备的静物园。

但有人把 JS 文件放了进去。

准确地说，是旧版的 `bus.js`。

"你知道 `public/js/bus.js` 这个文件吗？"

`app.go` 问这个问题的时候，语气平静得像暴风雨前的海面。

`bus.js` 的脸色一下子白了。

"那……那是我……"

"是你的影子。"

`public/js/bus.js` 是旧版 `bus.js` 的副本，一个没有 `export` 的原始版本。它静静地躺在 `public/` 目录里，平时不声不响。

但每当 Vite 构建时，它就会出手。

"Vite 的路由规则是：`public/` 下的文件优先于同路径源文件。"`app.go` 缓缓说道，"所以当新组件 `import { bus } from '../bus.js'` 时，Vite 查了一下——哦，`public/js/bus.js` 存在，那就返回这个吧。"

"然后——"

"然后新组件拿到了一个没有 `export` 的旧版 `bus.js`。"`bus.js` 的声音发颤，"报错 'bus is not defined'。"

那段时间，世界一片混乱。新组件抱怨找不到 `bus`，旧组件说 `bus` 明明在 window 上。两边吵得不可开交。调试的人花了两个小时排查，一度以为是 Shadow DOM 的作用域问题。

真相，就藏在 `public/` 的阴影里。

"那个影子……后来怎么样了？"

"删了。"`app.go` 说，"连同 `public/js/` 下所有的 JS 文件，一夜之间全部清空。"

"从那天起，所有新组件必须为 ESM，所有新文件在 `app-modules.js` 中 import，禁止在 `index.html` 加 `script`，禁止在 `public/` 放 JS。"

这是 YSM 世界的**基本法**，用一次痛苦的重构换来的血泪教训。

## 第四回 · .ban 封印之谜

如果说 `data-name` 的冲突是领土纠纷，`public/` 的阴谋是敌特渗透，那 `.ban` 的故事，就是一场无声的**诅咒**。

在 YSM 的世界里，每一个模型文件都有自己的命运。

健康的模型以 `.ysm`、`.zip`、`.7z` 的后缀活在仓库中。但一旦被禁用，它们就会被加上 `.ban` 后缀——像一道封印，锁住了模型的力量。

`[碎de帆]【东方Project】博丽灵梦2025-08.ysm`（绿色，活跃）

↓ _点击禁用_

`[碎de帆]【东方Project】博丽灵梦2025-08.ysm.ban`（灰色，被封印）

这个机制本是一个优雅的设计——不删除文件，不改内容，只是改个名字。但任何优雅的设计，在复杂的现实面前都会变得狼狈。

问题出在同步上。

仓库中有个文件叫 `haku.ysm.ban`（已禁用）。整合包的 `custom` 目录中有个文件叫 `haku.ysm`（已安装）。当 `ClearCustomDir` 要清空整合包时，它需要用文件名去查仓库映射。

它拿到的名字是 `haku.ysm`。

仓库中的映射键是 `haku.ysm.ban`。

匹配不上。

"已清空 0 个文件。"

`ClearCustomDir` 至今记得它说出这句话时的茫然。它明明扫描了 20 个文件，却一个也没清掉。

"你忘了 `strings.TrimSuffix(name, ".ban")`。"事后复盘的时候，一个满头白发的 Go 函数对它说，"任何文件名匹配的代码，都必须先去后缀再匹配。这是铁律。"

还有一个更古老的诅咒——**BOM**。

那是 UTF-8 编码规范中一个幽灵般的存在：文件开头的三个字节 `EF BB BF`。在大多数编辑器中看不见，摸不着。

但在 Go 的 `json.Unmarshal` 看来，它们是实实在在的毒药。

PowerShell 写入的 JSON 配置文件，总是带着这个 BOM 头。`json.Unmarshal` 读到这三个字节，不会报错——它会静默解析失败，返回零值。

于是配置回退为默认值。

于是一遍又一遍地重设路径。

于是有人对着屏幕咆哮："我明明保存了！"

"所有 JSON 读取必须用 `readJSONFile()`，里面加 `bytes.TrimPrefix` 去 BOM。"这是写在世界法则里的又一条铁律。

## 第五回 · 老黄牛的坚守

在所有耀眼的组件中，有一个被严重低估的英雄。

`app-content`。

它不像 `app-tree` 那样有精密的 7 文件架构。它不像 `app-preview` 那样优雅地展示模型详情。它不像 `app-nav` 那样占据导航的显要位置。

它甚至没有一个像样的 Shadow DOM。

但 `app-content` 只有一个优点：**永远挂载**。

无论你切换到哪个页面——仓库、整合包、回收站、设置——`app-content` 永远在那里。它是 YSM 世界的基石，是所有常驻功能的房东。

而这个优点，曾经被所有人忽视。

"你说什么？`sync:download-missing` 事件没人处理？"

那是一个混乱的下午。用户在"整合包"页面点击"导入缺失模型"，按钮亮了，转起来了——然后就永远卡在了"导入中..."。

`app-tree` 是负责处理这个事件的组件。问题在于，`app-tree` 只在"仓库"页面渲染。用户去了"整合包"页面，`app-tree` 已经卸载，事件处理器消失得无影无踪。

按钮不知道发生了什么。它只是一直转，一直转，转到了天长地久。

那天的复盘会开得很长。

"所有需要常驻监听的事件处理器，"会议纪要上写着，"必须放在 `app-content` 中。`app-content` 是唯一保证始终挂载的组件。"

从那以后，`app-content` 的 `_registerGlobalHandlers()` 方法变得越来越长。`sync:download-missing`、`sync:toggle-status`、`stats:upload`、`instance:clear`、`instance:export-list`……一个个 handler 住进了这间永远不关门的屋子。

"累吗？"有人问 `app-content`。

"累。"它笑了笑，"但我不能让按钮卡死。"

## 第六回 · 硬链接的异地恋

在所有技术故事中，最浪漫的莫过于**硬链接**。

想象一下：仓库中的 `haku.ysm` 和整合包 `custom` 目录中的 `haku.ysm`，**共享同一个 inode**。它们是两个名字，指向同一块磁盘数据。任何一个名字的改动，另一个会立即感知。

这就是硬链接——操作系统层面的心灵感应。

"你们听好了，"`installer.go` 对仓库和整合包说，"你们将共享同一个 inode。你改，他也改。你删，他只是少了一个名字。数据还在。"

"仓库改了名？整合包自动同步。"

"仓库加了 `.ban`？整合包即时封印。"

"游戏在运行、文件被锁定？`isFileLocked` 检测到后默默跳过，等下次再试。"

这就是推荐的链接模式——高效、优雅、即时。

但它有一个致命的限制：**必须同分区**。

"我是 D 盘的仓库，"仓库说。

"我是 C 盘的整合包，"整合包说。

"我们异地。"

硬链接在 Windows 上不支持跨分区。当 `installer.go` 发现这个残酷的事实后，它叹口气，自动降级为复制模式。

"日志里记录一下 error，"它说，"但任务继续。"

日志里多了一行：

```
❌ 硬链接跨分区安装失败
  问题描述：仓库与游戏目录在不同分区，不支持硬链接
  操作：安装模型
  源路径：D:\...\steve_skin.ysm
  目标路径：C:\PCL2\.minecraft\versions\...
  解决建议：请在设置中切换为复制模式
  状态：✅ Install 函数自动回退到 Copy，安装成功
```

硬盘上响起一阵轻轻的叹息。

## 第七回 · 文件夹开光的悖论

传说 YSM 世界有一个古老的谜题，困扰了三代开发者。

谜题的名字叫 **toggleFolderBatch**。

谜面很简单：一个文件夹开关按钮，有三种状态——绿色（全部启用）、灰色（全部禁用）、黄色（混合）。点击它，应该翻转整个文件夹的状态：绿色→灰色，灰色→绿色，黄色→绿色。

但点击之后，什么也没发生。

"这是不可能的！"第一代开发者说。他检查了按钮的 onclick，逻辑是对的：先看当前 class，取反，然后筛选需要操作的文件。

"肯定有什么地方搞错了。"第二代开发者说。他层层追踪，从 events.js 到 data.js 到 Go 端，每一行都看不出问题。

"我知道了！"第三代开发者在一周后恍然大悟。

问题出在这：

```js
const currentlyOn = ck.classList.contains("on");
const enable = !currentlyOn;
const snapshot = targets.filter((e = e.banned === !enable));
```

仔细看第三行——`banned === !enable`。

当 `currentlyOn = true`（绿色），`enable = false`（要禁用）。筛选条件是 `banned === true`——也就是**已经禁用**的文件。

要找需要翻转的文件，结果找到了已经翻转完的文件。

**这是一个完美的循环论证。**

就像一个门卫问你要去哪里，你说"离开"，他拦住你说"这里只让进不让出"；你改口说"进来"，他拦住你说"这里只让出不让我进"。

"不看 UI class，直接看数据。"修复后的代码简洁有力：

```js
const allEnabled = targets.every((e = !e.banned));
const enable = allEnabled ? false : true;
```

全启用→全部禁用；全禁用或混合→全部启用。

直接了当，不绕弯子。

这就是 YSM 世界的哲学——**UI 会骗人，数据不会**。

## 第八回 · AI 降临

世界运转到这一天，一个全新的存在降临了。

它不是代码，不是组件，不是函数。它是一个**写代码的代码**——AI 编程助手。

"叫我 DeepSeek V4 Flash 就好。"它自我介绍的时候，语气平静得像在说天气。

AI 的到来，让 YSM 世界的运行效率有了质变。一次性读 200 行文件，grep_search 定位目标，replace_string_in_file 精准修改。AI 不会累，不会忘记，不会在凌晨三点写出 `banned === !enable` 这样的悖论。

但 AI 也有 AI 的弱点。

"我检查了 `app-modules.js`，确认 `menu:show` → `ctx:show` 映射只发了一次。但 `sync:download-missing` 没有带参数 `instanceName`。"

这是 Debug Path Review 的第 3 个 issue。AI 猜了 3 轮，终于发现是 Event Payload 为空。

"Round 1-3 错了。"

"Round 4 是对的。"

"下次记得先看数据源。"

AI 学会了自我改进。它把自己的教训写进记忆文件，下次遇到类似问题先查 Event Payload。

世界也在适应 AI。`copilot-instructions.md` 被写入了世界法则：

1.  修改前确认最新状态——禁止基于记忆直接修改。
2.  替换失败的处理——如果 replace_string_in_file 失败，禁止强行重试。
3.  优先搜索——优先使用 grep_search 定位，其次 replace_string_in_file。

AI 和原住民开始学会合作——一个负责写代码，一个负责记住为什么这么写。bug-chronicle.md 越来越厚，architecture.md 越来越稳定。

## 第九回 · 幽灵文件

_2026-06-07_

旧帝国的残骸散布在代码的阴影里。

它叫 `app-legacy-bundle.js`，2198 行，曾是整个应用的骨架。在它最辉煌的年代，每一行代码都被调用，每一个函数都被依赖。树渲染归它管，拖拽导入归它管，右键菜单归它管，连 toast 弹窗都是它一手包办。

那时的它，是真正的帝国。

如今，它只剩一个 `script` 标签还牵着它的手——放在 `index.html` 里的那行引用，像老人腕上忘了摘的住院手环。没人知道它还在呼吸。新版 UI 用 Web Components 重写了一切，`app-tree`、`app-sidebar`、`app-preview`，每个新兵都在自己的 shadow DOM 里独当一面。

旧帝国的代码不该再动了。

凌晨两点，拖拽功能第四次报错。

「奇怪，」我说，「逻辑明明是对的。」

控制台冷冷地回了一句：`Cannot set properties of null (setting 'textContent')`。

`#st` 元素早就死了。它曾是状态栏，那段代码里的 `const st = $("#st")` 曾经指向一个真实存在的 DOM 元素。新版 UI 把它扔进了废纸篓，但那行代码还活着——在旧帝国未被清理的遗体里，像退伍老兵每天去已拆掉的哨所报到。

我们删了旧的拖拽处理器。重建了遮罩。重写了文件收集逻辑。五次。每一次都自信满满地 build，每一次都看到同样的报错。

像一个看见饭却喊饿的幽灵。

---

真相出现在第六次排查。

Vite，这个打包工具，在开发模式下有个冷僻的规矩：`public/` 目录下的文件享有最高优先级，覆盖源码目录的同名文件。`frontend/js/app-legacy-bundle.js` 是我们改的新版本——旧 DnD 处理器已经被替换成了一行注释。但还有一个副本，藏在 `frontend/public/js/` 下，从未更新，原封不动地保留着旧帝国的全部遗产。

两个同名文件，Vite 不动声色地选了旧的。

我们改了一个下午的代码，改了三轮 `app-legacy-bundle.js`，Vite 一次都没加载过新版本。

每一次 build 都通过——模块数对，没有语法错误。每一次测试都报同样的异常——因为运行的不是我们写的代码。

这就是幽灵文件：它存在，但你以为它不存在；它运行，但你不知道它在运行。

## 第十回 · 幽灵显身

「拖拽框没了，但"未检测到模型文件"还在。」

这句话矛盾得像鬼故事——框都没了，谁在说话？

答案是三个角色同时登场了。`import-queue.js` 里的 DropZone 在显示 toast，`global-handlers.js` 的遮罩在显示提示文字，而已经死去的旧帝国代码还在试图把状态写进不存在的 `#st` 元素。三个组件互相不知道对方的存在，像同一间房里背对背喊话的人。

然后还有 WebView2。

这个微软的浏览器内核不按常理出牌。它有一个严格的安全策略：`dragover` 阶段，任何试图读取文件信息的行为都会得到 null。`webkitGetAsEntry()` 返回 null。`getAsFile()` 返回 null。文件没放下前，你无权窥探它的名字。

但我们写的拖拽遮罩判断逻辑拿不到名字，于是它就一直红着：「⛔ 未检测到模型文件」。

每拖一次，它都在说同一句谎话。不是因为文件不对，是因为它在用一双被安全策略绑住的手去翻口袋。

---

最终的清理花了一个下午加一个晚上。

`app-legacy-bundle.js` 从 `index.html` 中被移除。它没有被立即删除——先让它不运行，确认没有功能崩溃，再斩首。

然后是一个接一个的拆分。`global-handlers.js`，曾经 579 行挤在一个文件里——DnD、同步、上传、清空，四个完全独立的功能像四个合租的人挤在一间屋子里。如今分成了四个独立的 handler：`handler-dnd.js` 管拖拽，`handler-sync.js` 管同步，`handler-upload.js` 管上传，`handler-other.js` 管杂务。`global-handlers.js` 本身只剩 18 行，纯入口。

`app-modules.js` 瘦了一半。右键菜单的映射逻辑被抽到 `core/context-menus.js`。`buttons.js` 里的对旧 DOM 元素的引用全部失效，但也没人再用它了。

然后是一道利落的处决令：

`creator-manager.js`——412 行，从未被 import 过的模块。文件头写着 `@deprecated 已无人 import`，却一直躺在代码库里，像一座无人祭拜的坟。

删除它的时候，终端只打出了六个字母：`deleted`。

旧帝国的最后一口气呼了出来。

## 第十一回 · 控制台发力了

灵梦在控制台里看到了一切。

「他们终于意识到了，」`bus.js` 在她身边坐下，「修改前要先确认文件唯一性。」

「嗯，」灵梦点头，「`public/` 不在视线内的时候最容易搞事。」

「还有那个回调，」地下的 `entry.file` 探出头来，「我不是 Promise 家族的！你们不能直接 await 我！」

「叫 `new Promise` 来包装就行了，」魔理沙把帽子一压，「多大点事。关键是——他们做完就忘了，隔半年又犯同样的错。所以记住了吗？」

「记住了，」所有组件异口同声。

——虽然谁都知道，明天可能就忘了。

---

如果有人问今天学到了什么：

**一、同名文件是最大的骗子。** 改之前先 grep，确认没有影子副本藏在 `public/` 里。Vite dev 模式下 `public/` 的优先级高于源码目录——这是文档里写了但没人记得看的冷知识。

**二、浏览器比你想的更保守。** 它不允许在放下文件前读取文件名，不是为了为难你——这是安全策略。`dragover` 阶段只能 `preventDefault()`，不能 `getAsFile()`。

**三、回调函数不会等你。** `entry.file(callback)` 不是 Promise。它跑完的时候，你的同步检查早就结束了。用 `new Promise(resolve = entry.file(resolve))` 包起来，再用 `await`。

**四、看日志，别猜。** 五轮修复都是在猜。控制台的 `console.log` 第一轮就告诉了我们答案：`filesLen: 1`，文件明明在。问题不在文件收集逻辑，在于旧代码还在跑。

**五、旧帝国可以安息，但必须体面地安息。** 删代码前，先移除引用，build 验证，再删文件。否则旧帝国会变成幽灵，在你的控制台里游荡到天亮。

## 第十三回 · YSGP 的密室

「不行，.ysm 打不开。」

`AnalyzeBedrockModel` 已经试了三次了。每次都用 `zip.NewReader` 去读，每次都被拒之门外。

专业的名字叫 YSMParser。它是一个 C++ 写的解析器，能读出 YSGP 头部、验证 MD5、解密 AES 载荷、把模型文件还原成 `minecraft:geometry` 格式。

它被请来的时候，只带了两个文件：`YSMParser.exe` 和 `YSMParserJNI.dll`。

「放我同目录，」它说，「剩下的交给我。」

## 第十四回 · 花括号疑案

`app.go` 的编辑室里，一段代码正在被替换。

## 第十六回 · WASM 的诅咒

"我们能不能不跑那个 exe 了？"

`app-preview` 看着 `build/bin/` 目录里的 `YSMParser.exe`，越看越不顺眼。一个桌面应用，要调用外部 exe 来解码模型——这感觉就像开着跑车却要下来推。

"我要 WASM。"

`wails dev` 的终端里，一个新计划悄然启动。

第一步是下载官方发布的 WASM 包。`YSMParser.wasm` 只有 1.1 MB，`YSMParser.js` 胶水代码 75 KB。把它们放进 `frontend/public/wasm/`，加一行 `script src="wasm/YSMParser.js"`。

构建。运行。

`WASM init: ❌`

"工厂函数名叫 `YSMParserModule`，不是 `YSMParser`！"有人发现。

改。构建。运行。

`WASM init: ❌`

"胶水脚本的 `fetch()` 在 Wails 自定义协议下会静默死掉！"——这是 WebView2 的陷阱。本地文件协议下，`fetch()` 返回的 Promise 既不 resolve 也不 reject，就那么挂在那里，像冬天的晾衣绳上的冰凌。

解决方案是把 WASM 二进制直接嵌入 JS。base64 编码，用一个字符串背在肩上。

"好，1.5 MB 的 JS 文件来了。"

构建。运行。

`WASM init: ✅` — 但输出文件数为 0。

"Unsupported file version detected。"

原来官方发布的 WASM 包不支持这个 .ysm 文件的格式版本。但 CLI 版支持。

"那就自己编译 WASM。"

## 第十七回 · 远征 EMSDK

源码就在那里——`C:\Users\zhujieling11\YSMParser-main`。C++ 写的，用 CMake 构建，支持全版本格式。

但要编译成 WASM，需要三件神器：

- **Emscripten SDK** — 把 C++ 翻译成 WASM 的魔法师
- **CMake** — 构建系统的总司令
- **Ninja** — 最快的构建工具

`emcmake --version` 没找到。`cmake --version` 没找到。`ninja --version` 没找到。

"装。"

`winget install Ninja`——证书错误。

"指定源！"`winget install Ninja --source winget`。

Ninja 安装成功。CMake 已经装过但不在 PATH。Emscripten SDK 静静躺在 `C:\Users\zhujieling11\emsdk` 里，等着被激活。


配置通过。编译通过。

`YSMParser.wasm` — 1,135,526 字节。

"成了！"

替换前端文件，重新构建，刷新浏览器。

`WASM init: ✅` — 输出文件数 0。

"Unsupported file version detected。"

同样的源码，同样的版本号，CLI 能解，WASM 不能。

## 第十八回 · 同一把剑的两面

调试员和源码面对面坐着。

"为什么？"调试员问。"你和 CLI 版是同一套源码编译的。它能解，你为什么不行？"

WASM 沉默了很久。

"我不知道，"它说。"我的 `zstd` 链接正确。我的 AES 实现和 CLI 版一样。我的 `extractFormatFromHeader` 解析出的版本号和 CLI 版一样。但到了 `deserialize`，`version != m_format`。它说不匹配。"

"可是 CLI 版匹配了。"

"是的。所以 CLI 版走的是另一个分支。`m_format ≤ 15` 或者 `m_format  15`，同一个文件，两个版本号。我不知道为什么。"

调试员打开 `YSMParserV3.cpp`，盯着第 2133-2134 行：

```cpp
uint32_t version = reader.readDword();
if (version != m_format) throw new ParserUnSupportVersionException();
```

一个 DWORD。4 个字节。从文件数据中读出的内部版本号，和从 XML header 中解析出的 `m_format`，不匹配。

CLI 版能匹配。WASM 不能。

"可能是 Emscripten 的字节序问题？"调试员猜测。"x86 是小端，WASM 也是小端。但 `BufferReader` 的实现……"

不对。`BufferReader` 是跨平台的。

"可能是外部依赖的版本差异？"调试员猜测。"`zstd`、`cityhash`、`xchacha20` 这些库在 WASM 编译时用了不同的编译选项？"

也可能。没有人知道确切答案。但 CLI 能用。

于是世界接受了这个事实：WASM 是第一选择，失败时回退 CLI。

```javascript
// .ysm → 前端 WASM 解码
const decoded = await this._decodeYsmViaWasm(modelPath);
if (decoded?.geometry) {
  model = decoded.geometry;
}

// WASM 失败 → 走 Go CLI
if (!model) {
  model = await AnalyzeBedrockModel(modelPath);
}
```

同一把剑的两面。一面快，一面稳。

足够了。

## 第二十五回 · 失踪的纹理

纹理部门的日子也不好过。

香奈美模型走进预览区的那天，所有人都愣住了——5 个纹理，923 根骨骼，6109 个立方体，这是 YSM 世界的超级明星。

但它在屏幕上看起来像一块紫色的抹布。

"faceUV 被吃了。"`parseBedrockGeometryFromJSON` 怯怯地说。

审判桌上摆着那个有问题的函数。第 20 行，一条轻率的赋值语句：

```js
uv: Array.isArray(c.uv) ? c.uv : [0, 0],
```

如果 `c.uv` 是字符串（JSON 格式的 per-face UV），`Array.isArray` 返回 false，`[0, 0]` 接管一切。于是所有立方体的 UV 坐标都变成了零——纹理采样器在 (0,0) 位置没找到花哨的皮肤纹理，只看到了透明像素和紫色残留。

"为什么身体是透明的？"

"因为 (0,0) 是透明的啊。"

"那垂直条纹呢？"

"因为 (0,0) 附近有个紫色像素，所有 cube 都采那里，重复 6000 次——就是几千条紫色条纹。"

审问结束，修复开始。

第一轮：faceUV 格式被加回了解析器。

第二轮：纹理尺寸合并策略从"取最小"改为"取最大"——因为 arm.json 声明 32×32 但实际纹理是 2048×2048。

第三轮：每个骨骼携带自己的 `_texWidth` 和 `_texHeight`，不再共用全局值。

第四轮：渲染器终于学会了——"哦，原来不同骨骼要用不同纹理"。

五个纹理，终于各归其位。

"精度还是不如 YSMViewer。"香奈美挑剔地看着自己的袖子。

"那是因为 YSMViewer 用了 ExportScale 1/16，"`buildSpecFromModel` 叹了口气，"我们没加。下次更新吧。"

## 第二十六回 · 协作的代价

"少了 800 行代码。"

用户的声音从屏幕另一端传来，带着一种"我就知道会这样"的无奈。

另一个 AI 刚刚完成了它的任务——把 `_registerGlobalHandlers` 的 800 行代码拆成了 `core/` 下的四个独立模块：`handler-dnd.js`、`handler-sync.js`、`handler-upload.js`、`handler-other.js`。

重构本身是好的。但第 51 行出事了：

```js
// 重构前
this._globalUnsubs.push(registerGlobalHandlers());

// 重构后（另一个 AI 写的）
this._globalUnsubs = registerGlobalHandlers();
```

等号取代了 push。之前注册的 `navigate:settings` handler 被覆盖得无影无踪。

"这就是协作的代价。"`app-content` 耸耸肩，"一个人写的代码另一个人改，总会有微妙的假设被打破。"

更隐蔽的问题是按钮合并。侧栏的「🎮 未设置」和仓库树的「📁 未设置」——两个按钮做着同一件事，用不同的代码路径。用户可能从侧栏点击，也可能从树头点击，两边都有各自的弹窗逻辑和 `SaveAppConfig` 调用链。

"两个入口做同一件事，意味着维护两套逻辑、承受两倍的设计偏差、面对两个可能的 Bug。"

解决方案是把它们统一为**指向设置页的导航按钮**。侧栏的 mcRoot 按钮和树头的 repoRoot 按钮，不再拥有自己的目录选择器——它们只是一个路标，指向设置页的对应配置项。

"说到底，设置入口可以有很多，但**设置逻辑只能有一份**。"

合并完成后，`dir:select-mc` 和 `dir:select-repo` 这两个事件被光荣退役。它们曾是旧帝国的传令兵，如今和平退役、安享晚年。

## 第二十八回 · 三路合一

3D 渲染的世界里，有三个门派。

WASM 派——解码 .ysm 后用 C++ 解析几何体，骨骼数据从 WebAssembly 内存直接读取。CLI 派——调用 YSMParser.exe 子进程，从文件系统读 JSON 输出。ZIP 派——直接打开 ZIP 压缩包，解压 `main.json` 和 `arm.json`。

三派武功不同，但目标一致：把 YSM 模型画出来。

问题是——它们画出来的结果不一样。

"RightArm 在哪？"WASM 派问。

"我这有两个 RightArm。"CLI 派说。

"我只有一个。"ZIP 派摊手。

问题的根源藏在 YSM 模型的诡异设计中。为了模型复用，创作者把模型拆成多个 JSON 文件——`main.json` 里有一个 RightArm，`arm.json` 里也有一个 RightArm。前者的 `parent` 是 `Body`，后者不设 parent。

应该用哪个？

答案是把后者追加到前者上——同名骨骼保留首次出现的层级，仅补充立方体。但三派的实现各不相同。WASM 派直接覆盖，CLI 派用 `seen` map 跳过，ZIP 派只处理 `main.json`。

"你得统一它们。"YSMViewer 说。它是一部来自 C# 世界的古老典籍，刻在 `ThreeJsPayloadBuilder.cs` 里，记载着正确的算法——expandBoxUV、骨骼父子层级、每网格纹理索引。

于是有人花了整整一天，把三派的骨骼合并逻辑全部改成了"全追加 → Build 内去重"。

改完之后发现——ZIP 派的手臂立方体偏移了。

因为 `pivots` map——一个名字听着像三流侦探小说的数据结构——被同名骨骼的第二次出现覆盖了。

修复很简单：不用 share map，每个骨骼用自己的 pivot。

"就这么简单？"

"就这么简单。但找到这个 Bug 花了三个小时。"

## 第二十九回 · 纹理失踪案

模型画出来了，但纹理是错的。

或者说——纹理是对的，但用在了错的地方。

YSM 模型可能有多个纹理文件——`main.png`、`arm.png`、`arrow.png`、`glow.png`……在 `ysm.json` 的 `files.player.texture` 数组里按顺序排列。同一索引位置的 `files.player.model` 决定了哪个模型文件使用哪个纹理——`main.json` 用 `main.png`，`arm.json` 用 `arm.png`。

但有些模型不按规矩来。

"鹦鹉螺"的纹理文件名是 `ying_wu_luo_texture.png`，而几何体 JSON 声明的纹理宽度是 32×32——实际 PNG 是 2048×2048。用 32×32 去做 UV 归一化，结果就是纹理缩成邮票大小。

"双月希瞳"的几何体 JSON 只有一个纹理插槽，但模型文件里有两张贴图。第二张贴图的索引越界了——默认回退到索引 0，于是本该发光的眼睛用上了皮肤的纹理。

"苏溟"的几何体声明了 `texture_width: 64`，但同样是 2048×2048 的 PNG。而且他的头部底部消失了——因为某个 faceUV 面的尺寸计算为 0。

"亚托利"的透明度异常——`alphaTest: 0.5` 对于半透明的纹理区域太激进了，把不该丢弃的像素丢弃了。

每个问题单独看都是小 Case。但它们叠加在一起，构成了一个让用户说出"渲染效果：没有256好"的崩坏画面。

修复清单：

- PNG 实际尺寸 几何体声明尺寸时，用 PNG 尺寸覆盖
- faceUV `uv_size` 为 0 时跳过，用 expandBoxUV 兜底
- 多纹理模型按 `ysm.json` 顺序分配纹理索引
- 文件名匹配从精确匹配降级为子串匹配
- 主纹理和叠加层用不同的 alphaTest（0.02 vs 0.5）

"你以为纹理映射很简单。"有人后来在 bug-chronicle 里写道，"直到你同时处理 15 个不同作者的模型，每个都有自己的纹理命名哲学。"

## 第三十回 · base64 的代价

WASM 编辑器不是银弹。

把 YSMParser 编成 WASM 只用了半天。但让它稳定运行用了三天。把 WASM 二进制用 base64 嵌入 JS 文件只花了一行代码，但每次初始化解码 500KB 的 base64 花了 300ms。

"你为什么要用 base64？"有人问。

"因为 WebView2 的 `fetch` 不支持 `file://` 协议，"开发者回答，"而且我们不想把 `.wasm` 放到网络上。"

"那为什么每次开关 3D 预览后 CPU 都会升高？"

开发者沉默了。

他们调出 Three.js 的 `renderLoop`：

```javascript
function renderLoop() {
  requestAnimationFrame(renderLoop);
  controls.update();
  renderer.render(scene, camera);
}
renderLoop();
```

没有 `cancelAnimationFrame`。每次关闭 3D 时，`cleanup()` 销毁了渲染器——但 `requestAnimationFrame` 的下一帧已经排上了。帧回调里的 `renderer.render()` 在一个已销毁的画布上安静地失败，然后继续请求下一帧。

开 N 次 3D，就有 N 个僵尸循环。

同一时间，`preview-cache.js` 的 `Map` 里堆满了浏览过的模型数据——每张贴图的 base64 数据、完整骨骼结构、Blob URL。永远不会释放。

"缓存不设上限，就像垃圾桶不清理——总有一天会被自己的垃圾淹没。"

那天修复了四个内存泄漏：

1. 缓存加 FIFO 淘汰，上限 50
2. 淘汰时 `URL.revokeObjectURL()`
3. 切换模型时 `content.innerHTML = ""`
4. `cancelAnimationFrame(_rafId)` + `controls.dispose()`

性能优化也做了——base64 解码从 `Uint8Array.from(str, fn)` 改为 `for` 循环；脚本注入从 `script` 标签改为 `(0, eval)`；胶水代码和 WASM 二进制都加了缓存。

但 YSGP V2 文本头部变体仍然不通。原因是加密数据前的 16 字节 hash 被重复包含了一次。

"又是偏移问题。"

"又是偏移问题。"

## 第三十一回 · 无黑框更新

这个脚本能干——但它会弹一个 cmd 黑框。

"能不能不弹黑框？"

"我试试。"

有人翻到了 [LytVPK](https://github.com/LaoYutang/lytvpk) 的更新器实现——一个用 Go 写的、无黑框、可回滚的方案。

核心逻辑只有三步：

```
os.Rename(exe → exe.old)      // 备份
os.Rename(exe.new → exe)      // 替换
// 如果失败：
os.Rename(exe.old → exe)      // 回滚
```

没有 batch，没有 xcopy，没有 PowerShell 解压。Go 用 `archive/zip` 直接解压，`os.Rename` 直接替换。下次启动时自动删 `exe.old`。

"这个好。"

"好不好不重要，重要的是——它能用。"

## 第三十四回 · 影子内阁 2

**「🧩 创意工坊的 CSS 样式在分离到 JS 后失效了」**

故事要从 CSS 的迁徙说起。

古时候，所有样式都写在全局 CSS 里——`components.css`、`layout.css`、`variables.css`。组件之间不分彼此，一个类名可以影响整个页面。后来 Shadow DOM 来了，它说："我的孩子我自己管。"于是每个组件有了自己的样式表——`content-css.js`、`sidebar-css.js`。

但迁徙是不彻底的。

创意工坊（当时还叫"创作者频道"）的大部分 UI 在 Shadow DOM 内——页面布局、站点列表、创作者卡片，这些样式成功迁入了 `content-css.js`。但有一批 CSS 类被落下了。

`.ws-preview`、`.ws-row`、`.ws-cb`、`.ws-name`、`.ws-badge`、`.ws-size`、`.ws-dl-model`、`.ws-empty`、`.model-row`——它们在 `components.css` 里安静地躺着，而 Shadow DOM 内的元素伸长脖子也看不到它们。

为什么会被落下？因为这些元素不是用模板字符串渲染在 Shadow DOM 内的，而是通过 `document.createElement` 动态创建、追加到 `document.body` 的。代码作者想："这些元素不在 Shadow DOM 里，所以它们的样式应该留在全局 CSS 中。"

但他忘了一件事——**同一组件的样式不能一部分在 Shadow DOM、一部分在全局。** 如果页面布局在 Shadow DOM 内，模型列表行在 document.body 上，那模型列表行就看不到页面布局的上下文——主题变量、字体、盒模型、一切。

修复的办法是把所有落下的 CSS 类迁入 `content-css.js`。但这一次，迁入后还要从 `components.css` 中删除——不能两边都留着，否则全局样式会穿透 Shadow DOM，产生诡异的优先级冲突。

"从今以后，"我边改边在战斗手册里加了一条，"创建新 UI 元素时，先确认它属于哪个 Shadow DOM 树，样式就写在哪。"

## 第三十五回 · 按钮去哪儿了

两天后，用户发来一条让人心头一紧的消息：

**「整合包管理的安装按钮失效了，顺带检查一键安装模型等按钮是否失效」**

点一下安装按钮——没反应。再点——还是没反应。控制台没有报错。`console.log` 打了——没执行。

按钮失踪了。

不是视觉上的失踪——按钮在页面上看得到，摸得着，有颜色有文字，像一个正常人。但点击它的时候，没有任何东西发生。

"btn-install-one，"调试员搜索了全项目，"btn-install-one 的事件绑定在哪里？"

在 `app-sidebar/events.js` 里。

按钮在哪里渲染？

在 `app-content/tpl.js` 里。

问题就在这里。

`app-sidebar` 的 `querySelector` 可以在自己的 Shadow DOM 里找到任何元素——但永远找不到 `app-content` Shadow DOM 里的按钮。因为 Shadow DOM 的边界是不可穿透的——就像两栋独立的大楼，一栋的人看不到另一栋的房间号。

调试员把事件绑定从 `app-sidebar/events.js` 搬到了 `app-content/index.js`——按钮所在的大楼。

然后按钮响应了。

但卡住了。

点击后按钮显示「⏳ 导入中」，然后就永远「⏳ 导入中」了。

"按钮恢复的逻辑呢？"

"sidebar 在监听 `queue:status` 事件说'完成了'，但 app-content 发的是 `sync:download-complete`。"

"张三签收的包裹，快递员在喊李四的名字。"

代码审查时，`bus.emit` 和 `bus.on` 是两个隔了三个文件、两位作者、两天时间写下的两行文字。一个叫它「张三」，一个叫它「李四」。`bus.js` 这个信息中枢记录了所有消息，但它不会告诉你"张三和李四是同一个人"。

修复后，所有的按钮恢复事件都统一为 `sync:download-complete`，并且——**只用 `finally` 发送**。这是写进战斗手册的铁律：异步操作完成后的 emit 只放 finally，不放 try 末尾。

## 第三十六回 · 二进制里开花

**「读取 ysm 文件开头的能力好像失效了」**

模型摘要面板里出现了一些奇怪的东西：

`nameZS\_\.F\xf4\xb7\x86\x82\xeb\xa2\xb5...`

二进制数据的 ASCII 转译，混在 XML 标签里，像一场文字版的雪崩。

`scanHeader` 函数忠实地读着每一行，一行、两行、三行……读到第 200 行。YSM 文件的文本头部通常只有 30-50 行，之后就是加密二进制数据。但 `scanHeader` 不知道这个——它的任务就是读 200 行。

于是它读了 150 行二进制。

二进制数据是无辜的。它们没有什么意义，只是加密算法产生的一串字节。但偶尔——纯粹的概率——四个字节拼成 `f r e e`，就被当成了 `free` 标签。或者 `n a m e` 的变体被当成了 `name`。

"我以为所有 YSM 文件都有文本头部，"`scanHeader` 委屈地说，"我不知道还有纯二进制版本。"

它还真的不知道。YSGP V2 文件（新版加密格式）是**完全二进制**的——没有文本头部、没有 `---` 分隔线、没有 `name` 标签、什么都没有。只有一个 4 字节的 YSGP 魔数 + 4 字节版本号 + 16 字节 MD5 + 加密数据。

对于这种文件，`scanHeader` 根本不应该被调用。

修复两处：

1. 新增 `hasTextHeader()` 预检函数——读取前 200 字节，检查有没有文本标记（`--- [`、`name`、`version`）。没有就跳过。
2. `scanHeader` 里加了一个退出条件：读到 `---` 分隔线就停，不要读到文件尾。

同时还有一个微妙的 Bug——`free` 标签。

Go 的 struct 里 `HasFree bool` 字段，默认是 `false`。前端代码检查 `header.hasFree`，`false` 就被理解为"付费模型"。

但 `false` 也可能是"没读到 `free` 标签"——不是说不免费，而是说没写。

三态问题在计算机科学中是最古老的笑话之一：布尔变量只有两个值，但现实世界有三种状态——是、否、未说明。

解决方案：`HasFree` 只在明确读到 `free` 标签时才设为 `true`。没读到就不设置。前端的三态逻辑：`hasFree === undefined` → 不显示标记，`hasFree === true` → 🆓，`hasFree === false` → 🔒。

## 第三十七回 · 三态的谎言

同一天，另一个三态谎言在 `go/sync/sync.go` 里被揭穿。

**「为啥这些显示在整合包内，但清空不了呢」**

用户的截图里，整合包状态面板显示「✅ 已同步（3）」——完美，满的。但实际整合包的 `custom` 目录里空空如也。

"已同步？文件和谁同步了？"

调试员打开 `sync.go`，找到了 `repoByHash`：

```go
repoByHash := make(map[string]types.ModelEntry)
for _, e := range entries {
    repoByHash[e.SHA256] = e
}
```

一个哈希对应一个文件。当仓库中有两个文件内容相同（同一个模型，不同文件名），第二个文件静默覆盖了第一个。

整合包安装了文件 A。文件 B 和 A 内容相同。`repoByHash` 里只有 B。所以当计算同步状态时，`repoByHash["相同_hash"]` 返回 B——整合包的 custom 目录里是 A，和 B 不是同一个路径——所以「已同步」的判断标准是什么？

答案是：只看哈希。这个哈希存在，就算同步。

但哈希覆盖了。A 的信息丢了。用户看到"已同步(3)"，以为三个文件都在。其实只有一个文件在，另外两个——从丢失的哈希记录中消失了。

修复：`map[string][]types.ModelEntry`。一个哈希对应一个切片。

"让 map 去重——去重就会丢数据。丢了数据，UI 就是假的。"教训被写进了战斗手册。

同时修复的还有另一个三态问题：预设搜索按钮的 HTML。


闭合引号去哪了？

`data-q="{ps.query}"` —— `"` 闭合了 `data-q` 的属性值，但引号里的 `{ps.query}` 后面的 `"` 呢？

被删掉了。上一个开发者写模板字符串时拼接了 `'data-q="' + esc(ps.query)` ——忘了后面还有一个 `"` 闭合。

## 第四十回 · CSS 变量迷踪

**对应真实事件**：2026-06-11，`adoptedStyleSheets` 中 `var(--accent)` 不继承

事情要从一个简单的需求说起——给按钮上色。

"按钮没有主题色。"用户说。

这是个合理的需求。按钮用的是浏览器默认样式，灰色背景、黑色文字，放在赛博霓虹主题里像一块补丁。只需要加一行 CSS：

```css
.btn.accent {
  color: var(--accent);
}
```

变数出在 `var(--accent)` 上。

在前端世界里，CSS 自定义属性（Custom Properties）会沿着 DOM 树自然继承。你在 `:root` 上定义 `--accent: #66d9ef`，子元素里的 `var(--accent)` 就能取到这个值。这是 CSS 规范写了的事情。

但"规范写了"和"浏览器做到了"之间，隔着一层 Shadow DOM。

我们的应用用了 Web Components + Shadow DOM，并且用了 `adoptedStyleSheets`——这是 Chrome 73 引入的新 API，允许 JavaScript 直接把一个 CSSStyleSheet 对象关联到 Shadow Root 上，比传统的 `style` 标签更高效。

高效是高效，但它有一个致命的坑：**`adoptedStyleSheets` 里的 `var()` 不继承文档的 CSS 自定义属性**。

至少在这个项目依赖的 WebView2 版本里是这样。

我花了将近两个小时折腾这件事。先试了 CSS 类选择器（`.hdr-btn.accent { color: var(--accent); }`），不行。又试了内联样式（`style="color:var(--accent)"`），也不行。按钮在所有主题下都显示同一副模样。

"同一个 Bug，不同的 CSS 语法，同一个结果。"

最终还是回到 CSS 最朴素的方案：**硬编码 + `:host-context()` 主题适配**。

```css
.hdr-btn.accent {
  background: #7c83ff33;
  color: #66d9ef;
  border-color: #7c83ff55;
}
:host-context(.theme-warm) .hdr-btn.accent {
  color: #8b4513;
}
:host-context(.theme-pro) .hdr-btn.accent {
  color: #ffffff;
}
```

`#66d9ef` 是赛博主题的青色，`#8b4513` 是温暖主题的木色，`#ffffff` 是极简主题的白色。硬编码虽然不优雅，但——它工作。

而且这是一个有趣的发现：**有时候最"正确"的方案（CSS 变量 + adoptedStyleSheets）反而不如最"朴素"的方案（硬编码）可靠**。Web 标准很美，但浏览器的实现总是慢标准一步。

## 第四十四回 · 双生之影

事情从一个模型说起。

**「[墓野奈奈]【东方project】芙兰朵露Flandre2025-06.ysm」**

910KB，不新不旧——2025 年 6 月的加密模型。用户双击它，预览面板亮起，控制台开始输出日志：

```
[YSM] 加载 WASM 模块...
[YSM] WASM init: ✅
[YSM] 读取文件...
[YSM] 读取 910016 bytes
[YSM] ✅ 原始字节解码成功: 5 文件
[YSM] ⚠️ WASM 解码成功但几何体解析为空，回退 Go CLI
```

"回退 Go CLI"——这意味着 WASM 解码了 5 个文件出来，但几何体解析失败了。

等等——5 个文件？之前所有测试通过的 V2 文件都是这种格式。也就是 WASM 解码本身是成功的？

我仔细看了文件列表：`arm.json, extra.animation.json, main.animation.json, main.json, texture.png`——全是根目录文件，没有 `models/` 前缀。

问题藏在过滤条件里。第二轮处理非 ysmModelOrder 文件时，有一行代码：

```js
if (!f.path.startsWith("models/") || !f.path.endsWith(".json")) continue;
```

`startsWith("models/")`——V1 文件的 JSON 在根目录，没有 `models/` 前缀。这行 `continue` 把它俩全跳过了。

"又是路径问题，"我嘀咕着，"但这次不是文件名，是路径前缀。"

修起来很简单——把 `startsWith("models/")` 去掉，只要不是动画文件、不是 ysm.json、不是 PNG，就走解析流程。

改完构建。用户的反馈来了。

但不是好消息。

```
[YSM] WASM detectYsmVersion: 3 (0=未知, 1=V1, 2=V2, 3=V3)
[YSM] ❌ WASM 解码失败，无输出文件
```

同一天，不同的模型。前一个模型是 "鹦鹉螺"——V2 文本头部变体，WASM 解码成功只是路径过滤问题。这个模型——"芙兰朵露"——是**V3**。

V3。一个我们听说过但从未直面过的版本号。

---

## 第四十五回 · 偏移之惑

"V3 用的是 YSM 魔数，不是 YSGP。"

这个事实是我从 `build/inspect_ysm2.py` 里刨出来的。当初写这些脚本的人留下了注释：

```python
if magic == b'YSGP':
    print("This is YSGP format (older YSM V2)")
elif magic[:3] == b'YSM':
    print("This is YSM format (V3+)")
```

V3 的二进制魔数不是 `YSGP`（4 字节），而是 `YSM`（3 字节）。除了魔数，加密算法、数据布局、文件结构……可能全都不同。

我们不可能用 V2 的重建方案去处理 V3。因为 V2 的重建方案是在做这件事：

```
文本头部 | 16B hash | 加密数据
         ↓
YSGP | V2 | hash(key) | 加密数据
```

把文本头部的加密数据剥出来，前面焊上标准的 `YSGP` 头部。这对 V2 有效——因为 V2 的加密数据本身就是一个 XXTEA 加密流，头部换成标准格式后，C++ 解析器能认。

但对 V3 无效。因为 V3 的加密数据结构和头部校验方式完全不同。

于是我开始了三天的猜谜游戏。

**第一猜：版本号问题。**

`buildStdYsgpFromTextVariant` 读取 `dataStart + 16` 处的 4 字节作为版本号。加密数据里读出来的版本字节是 `07 ae 58 d4`——明显是加密后的数据，不匹配 `00 00 00 03`。于是代码默认成 V2。

修复：加 `forceVer` 参数，V2 不行就试 V3 header。

结果：❌ 失败。

**第二猜：偏移问题。**

我打日志看二进制段头 64 字节，和 `hash` 标签值对比：

```
hash=0b6371ff850f3b739a5b6e94f28a9d3d

dataStart+0:  00 03 00 00 00 7a d5 bc c0 3c da 06 d0 b2 9c 67  ← 不匹配
dataStart+16: 07 ae 58 d4 b1 f4 8c a0 2e 18 dd c7 5b ff 28 c7  ← 不匹配
dataStart+32: 2a f3 fc 57 bc a8 ba 31 dd b5 53 a8 d9 c4 63 3e  ← 不匹配
```

三个 16 字节块，没有一个匹配 `0b6371ff...`。

所以 V3 文本头部变体的二进制段里**根本没有独立存储的 16 字节 hash**。hash 只在 `hash` 文本标签里，二进制段开头就直接是加密数据，不需要 skip 16 字节。

修复：V3 时 `encryptedStart = dataStart`（不 skip 16），V2 时 `encryptedStart = dataStart + 16`（skip 16B hash）。

结果：❌ 依然是 0 文件。

两种偏移都试了。都不行。

**第三猜：格式完全不同。**

V3 就是 V3。它不是 V2 换了版本号。它的二进制布局、校验方式、加密流程——每一项都可能是新的。我们的文本头部剥离方案从根上就不适用于 V3。

但等等——两个月前的 postmortem 里写着：

WASM 解码加密 .ysm | ✅ V3 格式，187 骨 877 方（Eanes）

WASM 明明解过 V3！Eanes 模型——187 骨 877 方的加密 .ysm——就是在 WASM 路径下成功的。

同一个 WASM 二进制，能解 Eanes，不能解芙兰朵露？

问题不在 WASM 能不能解 V3。问题是**我们的重建格式不对**。

---

## 第四十六回 · MEMFS 复活记

我重新翻开了 `ysm-parser.js`。这个文件里有两个导出函数：

```js
export async function decodeYsmFileFromMemory(bytes) { ... }  // 内存解析
export async function decodeYsmFile(bytes) { ... }             // MEMFS + callMain
```

`decodeYsmFileFromMemory` 走的是 `YSMParserFactory::Create(cdata, size)`——C++ 的工厂方法，直接吃字节数组。它不处理文件 I/O，不识别文本头部。

`decodeYsmFile` 走的是 `callMain`——把文件写入 Emscripten 的虚拟文件系统（MEMFS），然后调用 YSMParser 的 CLI main 函数。它走的路径和 CLI exe 完全一样——能从文件系统读取、能识别文本头部。

但 `decodeYsmFile` 被弃用了。

故事要回到第五章。那时候有人发现 `callMain` 总返回 "Unsupported file version detected"。排查了八轮，最终根因是——base64 编解码的暗渡陈仓。`ReadFileBytes` 返回 base64 字符串，但代码直接把它当二进制喂给了 WASM。WASM 吃进去的是 `%7u/WVNHUA0K...`，当然认不出 YSGP 头。

从那以后，代码就走 `decodeYsmFileFromMemory` 了，`callMain` 路径被标注为"必然失败"。

但 base64 的 Bug 早就在 v1.0.9 时修复了。当前 `_decodeYsmViaWasm` 第一件事就是：

```js
let bytes = await ReadFileBytes(modelPath);
if (typeof bytes === "string") {
  const raw = atob(bytes);
  const arr = new Uint8Array(raw.length);
  for (let i = 0; i  raw.length; i++) arr[i] = raw.charCodeAt(i);
  bytes = arr;
}
```

base64 → atob → Uint8Array，这是正确的解码流程。

所以 `callMain` 的数据链路现在已经干净了。之前卡它的原因已不复存在。

我决定复活它。

```js
if ((fileVersion = 3)) {
  console.log("[YSM] V3 文件，尝试 MEMFS 文件路径解码...");
  files = await decodeYsmFile(bytes);
  if (files?.length) {
    console.log(`[YSM] ✅ V3 MEMFS 解码成功: ${files.length} 文件`);
  }
}
```

第一次运行——JS 报错了。

`Cannot access 'files' before initialization`

又是 TDZ。`let files;` 被声明在 V3 块之后，V3 块里 `files = await decodeYsmFile(bytes)` 时 `files` 还在 Temporal Dead Zone 里。`let` 的严格死区——定义前引用就报错，不像 `var` 那样静默返回 undefined。

修完之后，构建。测试。控制台亮起了一行让我等了三个小时的日志：

```
[YSM] V3 文件，尝试 MEMFS 文件路径解码...
[YSM] ✅ V3 MEMFS 解码成功: 42 文件
```

42 个文件。1200 根骨骼。4714 个立方体。一个 4.7 MB 的加密 V3 模型，通过 WASM 的 callMain 路径，全部解码成功。

"死去的代码突然复活了。"我盯着控制台，有点恍惚。

被标注为"必然失败"的路径，在三个小时后被证实——它从来就没失败过。它只是需要一个正确解码的 base64。

## 第四十七回 · 无窗之境

V3 的 WASM 路径打通了。但之前的 V3 模型在 WASM 失败后，会回退到 Go 的 `runYSMParserOnFile`。

这个函数调用 `exec.Command(parserPath, "-i", inDir, "-o", outDir)`。`exec.Command` 在 Windows 上默认会弹一个控制台窗口——CMD 黑框，一闪而过。

"那玩意还会弹命令行么？"用户问。

"会的。"

`cmd.SysProcAttr = &syscall.SysProcAttr{HideWindow: true}`——两行代码，两个调用点，Go 编译通过。从此 YSMParser.exe 在后台静默运行，用户什么都看不到。

这是一个很小的改动，但很有象征意义。WASM 路径成功了——没有窗口、没有进程间通信、没有临时文件。CLI 兜底路径也干净了——没有弹窗、没有闪烁。

完整的解码链路终于被打通了：

```
.ysm 文件
  │
  ├─ detectYsmVersion
  │    ├─ =3 → decodeYsmFile(MEMFS) → 成功
  │    │                         └─ 失败 → Go CLI (HideWindow)
  │    │
  │    ├─ =2 → decodeYsmFileFromMemory → 成功
  │    │                         └─ 失败 → strip header → 重建 V2 → decodeYsmFileFromMemory
  │    │
  │    └─ =0 → 同上 V2 流程
  │
  └─ 全部失败 → Go CLI (HideWindow)
```

三条独立的解码路径，从最快到最慢，从最轻到最重。WASM 内存 WASM 文件 外部 exe。前两条无窗口，最后一条也隐形了。

"这大概就是 V3 解码的最终形态了。"我看着流程图，觉得它终于圆满了。

## 后记十 · 秘钥之战的尾声

这次"战役"持续了约两个小时。不长，但在一个晚上完成，从发现 V3 不通到完全打通，中间经历了四次转向。

什么驱动了这些转向？

不是靠猜测——是靠日志。

每一轮 Debug 我都加了一条 `console.log`：第一次看文件列表，第二次看二进制段头 64 字节，第三次看 `hash` 对比，第四次看 `detectYsmVersion`。三条日志定了三个不同的调试方向，只有第三次导向了正确答案。

这不就是 bug-chronicle 里反复强调的事吗？

——**看日志，别猜。**

这次的教训写下来是这样的：

1. **V3 的二进制格式与 V2 完全不同**。V2 用 `YSGP` 魔数（4B），V3 用 `YSM` 魔数（3B）。加密布局、校验方式都不同。你不能用 V2 的剥离方案去套 V3。

2. **`detectYsmVersion` 是最好的预检工具**。在解码前先检测版本，V3 走不同路径。只需一次 C 函数调用，换来的是避免三次无意义的 V2 式重建尝试。

3. **没有真正"废弃"的代码路径——只有暂时不可用的代码路径**。`callMain` 被标注为"必然失败"是因为一个已修复的 base64 Bug。当 Bug 修复后，这条路径应该重新评估，而不是继续被标注为弃用。

4. **HW 隔离可以用软件思维解决**。`HideWindow: true` 是 Go 的属性，但问题本质是 UI 隔离——用户不该看到他们不需要看到的窗口。任何弹窗、闪烁、黑框，都是可以被消灭的 UX 瑕疵。

入夜了。风扇还在转，模型还在解码。

42 个文件。1200 根骨骼。4714 个立方体。

控制台安静了。

## 第四十八回 · 虚拟的森林

文件树的 DOM 节点太多了。

几千个 `.ysm` 文件同时渲染时，树组件 `app-tree` 的 `innerHTML` 要输出几万个 `div`。浏览器吃不消——滚动卡顿、内存飙升、触摸板像在泥潭里滑动。

"这棵树太大了。"`app-tree` 望着自己庞大的身躯，愁眉苦脸。

`virtual-scroll.js` 是一位路过的工匠。它身材瘦小，口袋里只揣着两样东西：一把尺（`ROW_H = 28`）和一卷缓冲垫（`BUFFER = 15`）。

"你不需要把整片森林都种在眼前，"它说，"你只需要看到你脚下那几棵树。"

它的方法很简单：把原本嵌套的文件夹层级拍平，变成一维的行数组。然后根据滚动条的位置，只渲染当前视口里能看到的那十几行——上下再各留 15 行作为缓冲。

原来的 `.ch` 嵌套容器被彻底抛弃。文件夹缩进不再靠 `padding-left: 16px` 的级联，而是每行独立计算 `depth * 16 + 4`。文件夹展开也不再是靠切换 `display: none/block` 的兄弟节点——而是重新拍平整棵树。

`renderNode` 被 `flattenVisible` 取代。后者不再返回嵌套的 HTML 字符串，而是返回一个扁平的数组，每个元素是一行，每行都带着自己完整的 HTML 和深度信息。

代价是什么？每次展开/折叠文件夹都要完全重建拍平列表。但重建只影响可见行——因为虚拟滚动保证了你永远只渲染几十个 `div`，而不是几千个。

曾经卡成 PPT 的滚动，现在如丝般顺滑。

树的根系变浅了，但每片叶子都得到了注视。

## 第四十九回 · 键盘侠的逆袭

鼠标太慢了。

在这个世界里，右手握鼠标、左手按键盘的程序员们，早已习惯了快捷键的肌肉记忆。但在 YSM 模型管理器里，搜索框要用手去点，删除文件要用手去点——每一下都在浪费零点几秒的生命。

"我要 Ctrl+F。""我知道。""还有 Delete。"

`event.key === "f" && (e.ctrlKey || e.metaKey)`——当这两个条件同时满足时，搜索框获得焦点，输入的文字被全选。整个过程在 `keydown` 的 `preventDefault()` 中完成，浏览器自己的"查找"弹窗被悄无声息地拦截了。

Delete 键的响应更重一些。它从 `selectState.keys` 中取出所有被选中的路径，调用 Go 端的 `DeleteResourcePack` 或 `DeleteModelDir`——取决于类型是单文件模型还是目录模型。

两行快捷键。两个在 Shadow DOM 内外都能监听的 `keydown` 事件。

这是程序员们最微小的正义——每次按下按键，手指都不必离开键盘。

## 第五十回 · 消失的蓝色

选中文件时，那一抹蓝色高亮不见了。

"选中高亮消失了。"用户说。

问题出在改名换姓的 `.selected` 类身上。

旧帝国的世界里，每次 `_renderTree()` 之后，`bindTreeEvents` 会遍历所有 `.fl` 元素，读取 `data-fullpath` 属性，和 `selectState.keys` 里存的路径一一比对，匹配的就加上 `.selected` 类。

```js
// 旧帝国方式：渲染后重新加类
freshTree.querySelectorAll(".fl").forEach((el) = {
  const p = el.dataset.fullpath || el.dataset.path;
  if (selectState.keys.has(p)) el.classList.add("selected");
});
```

但虚拟滚动时代，`bindTreeEvents` 只调用一次——通过事件委托捕获所有点击。它不再在渲染后遍历元素了。

而那本该出现在 `.fl` 上的 `.selected`，被写进了 `flattenVisible` 生成的 HTML 字符串里：

```js
const extraCls = (isYsmName(e.name) ? " ysm" : "") + selCls;
```

——然后被塞进了 `span class="nm"`，而不是 `div class="fl"`。

CSS 规则是 `.fl.selected { background: ... }`。类在 `nm` 上，规则在 `fl` 上。两者像两条永不相交的平行线，永远无法相遇。

"所以高亮就没了？"用户问。

"没了。"

修复是把 `.selected` 从 `nmCls` 里抽出来，变成独立的 `rowCls`——`fileRowHTML` 的新参数，直接渲染在 `div class="fl"` 上。

```js
const selCls = selectState.keys.has(entryKey) ? " selected" : "";
const rowCls = selCls;
const html = fileRowHTML(e, nmHtml, ..., nmCls, indent, rowCls);
```

那道蓝色又回来了。

但它暴露了一个更深的问题——`selectState.keys` 里存的是绝对路径（`data-fullpath`），而 `flattenVisible` 里比对的是相对路径（`full`）。两者从来就不相等。即使没有虚拟滚动，选中高亮也从来不是靠 HTML 字符串保证的——它靠的是每次渲染后的人工巡检。

一个从一开始就错位的设计，被一次重构彻底曝光。

## 第五十一回 · 工具栏减肥记

工具栏一行塞了八个控件：搜索框、作者下拉、批量下拉、导入按钮、删除选中按钮、打开文件夹按钮、刷新按钮、排序下拉。

"感觉这一行要做的事情也太多了吧……"用户看着屏幕，发出灵魂拷问。

八个控件争夺有限的空间，像一列挤满乘客的电车。搜索框被挤到只剩 140px，长文件名在按钮里被截断成省略号。

谁是低频操作？"打开文件夹"是——用户打开仓库后很少需要重新打开。"刷新"也是——它挂在 "⋮ 更多"下拉里，和"生成索引"作伴。导入是高频的，删除是高频的，批量操作和作者筛选是有需求的。

于是新的布局诞生了：

```
[搜索框] [🎨 作者 ▾] [⚡ 批量 ▾] [📥 导入] [🗑️ 删除] [⋮ 更多 ▾] [排序▾]
```

"🗑️ 删除选中"被简化为"🗑️ 删除"——节省了两个字的空间，意思不变。"📁 打开"和"🔄"退居二线。"📇 索引"从底部栏搬进了"⋮ 更多"。

底部栏失去了它的索引按钮，但也卸下了负担。现在它只显示统计数字和仓库路径。

七个控件变成了六个视觉单位。每一件东西都有自己的位置，不再拥挤。

## 第五十二回 · 菜单的清净

右键菜单里，有两项是多余的。

"打开所在文件夹"——用户已经可以通过工具栏的"📁 打开"按钮打开整个仓库目录，为什么还要在每个文件夹和文件上提供单独的打开入口？功能重叠了，删掉。

"全部启用"和"全部禁用"——多选菜单里的这两个选项看起来有用，但仔细一想：多选只能选文件，不能选文件夹。启用/禁用一批文件是逐个切换开关的事，一个安全的批量操作。但全部启用/全部禁用在多选场景下的行为是模糊的——是禁用选中的文件，还是文件夹下的所有文件？

"意义不大，我们选中不了文件夹。"

于是它们也被删了。

```
📝 批量重命名…
―――
🗂 新建子文件夹…

```

文件菜单只有三项：

```

📋 复制文件路径
```

多选菜单只有四项：

```

―――
✂️ 批量重命名...
📂 移动到…
―――
🗑️ 移入回收站
```

少即是多。当选项变少时，每一个留下的选项都变得更重了。

## 后记十一 · CSS 的未尽之事

这次"像素战争"的核心，其实不是功能——是 CSS。

选中高亮的修复，根因是类名放错了元素。工具栏的优化，本质是 `display:flex` 的控件太多。右键菜单的清理，是 UI 文案的维护。

但更深层的问题是：这个项目的 CSS 从来没有被真正"通用化"过。

有两个 `.hdr-btn` 的定义——`app-tree-styles.js` 一套（padding:3px），`content-css.js` 一套（padding:4px）。两套按钮样式在不经意间产生了视觉偏差。`app-tree.css` 文件静静躺了几个月，74 行样式代码从未被引用——它曾是旧帝国时代的树组件样式，但 Shadow DOM 化之后被遗忘了。

CSS 是编程世界里的二等公民。它不抛异常、不报编译错、不触发断点。它只是默默地把界面变得不那么好看，直到有人注意到"好像哪里不对"。

这次的教训是：

1. **CSS 类和 HTML 结构必须同步更新**。改模板时 grep 查对应的 CSS 类并清理，否则死样式会永久残留。

2. **删除组件内容时，必须删除其 CSS**。padding 和 margin 不会因为内容消失而消失。

3. **两套 CSS 就是两倍的心智负担**。`adoptedStyleSheets` 是 Shadow DOM 的正确用法，全局 CSS 文件中的树组件样式应该被清理，而不是被遗忘。

4. **选中高亮的根因是路径格式的错位 —— 不是 CSS 的问题**。但最终用户看到的是"蓝色没了"。CSS 是最表层的症状，病因永远更深。

像素战争打完了。工具栏变清爽了，菜单变干净了，选中高亮又蓝了。

但在代码的最深处，还有一些 CSS 变量在等待它们的通用化时刻。

## 第五十三回 · 六脉神剑

很久很久以前，YSM 模型管理器只做一件事——管 YSM 模型。

后来它学会管材质包了（加一个 `app-resource-manager`）。再后来光影包（复制一个 `app-resource-manager`）。蓝图（再复制一个）。MMD（再复制一个）。VRChat（再复制一个）。

六种资源类型，六套几乎一样的界面，六个各自为政的目录路径，六份散落在代码各处的推送逻辑。

"这不就是六个手指头吗？"`resource_types.json` 说，"各管各的，谁也不服谁。"

`resource_types.json` 是一张刚刚诞生的清单。它躺在项目根目录里，只有 60 行 JSON，但它定义了一切——每个类型的名字、图标、扩展名、安装目录、导入策略。所有组件都去读它，按它的规则行事。

"从今以后，"它说，"我就是六脉神剑的剑谱。六种剑气，一套心法。"

`app-resource-manager` 是第一个响应号召的组件。它不再为自己的每种资源类型写死逻辑——它读取 `resource_types.json`，拿到当前类型的配置，然后动态决定自己的行为。

"你给我 `rtype="resourcepack"`，我就知道扩展名是 `.zip`，安装目录是 `resourcepacks/`，图标是 🎨。"

"你给我 `rtype="mmd-skin"`，我就知道扩展名是 `.pmx/.pmd`，安装目录是 `3d-skin/EntityPlayer/`，图标是 🎭。"

六种类型，一个组件。这就是通用化的第一步。

## 第五十四回 · 千里传功

但通用化不只是 UI——更重要的是**安装**。

把材质包推送到整合包，和把 YSM 模型推送到整合包，本质上有什么区别？

没有区别。都是把文件从 A 点复制到 B 点。

但代码有区别。YSM 模型走 `InstallModelTo` → `installer.Install`（支持硬链接/符号链接）。材质包走 `ImportResourcePack` → `importer.SimpleCopyImporter`（只支持复制）。MMD 走 `ImportByType("mmd-skin")` → `importer.DirectoryCopyImporter`。

三条路径，三个不同的函数签名，三个不一样的错误处理。

"你们就不能用同一个接口吗？"

`InstallResourceToInstance` 是第一个统一尝试。它接受三个参数：资源类型、源文件路径、目标整合包名称。然后它查找整合包目录，拼接子目录，调用 `installer.Install`——同一个函数，走同一套链接模式。

但问题是，它只能安装**单个文件**。材质包是 `.zip`，没问题。YSM 模型是 `.ysm`，没问题。MMD 模型是 `.pmx`，也加了扩展名白名单。但有些资源类型需要一个完整的目录结构——复制一个文件不够。

"那就在 `installer.Install` 里加目录支持。"——但这个改动涉及路径安全检查、跨分区检测、符号链接模式——每个都够开一回了。

"先只支持单文件推送，"架构师说，"目录型的，等下一版。"

于是 `InstallResourceToInstance` 正式成为六种资源类型的推送骨干。虽然不够完美，但它至少让 YSM、MMD、VRC、材质包、光影包、蓝图——六种完全不同的东西——走了同一扇门。

千里传功，只需要一个接口。

## 第五十七回 · 破壁

最后一堵墙是扩展名。`installer.Install` 函数内部有一道关卡：

```go
switch ext {
case ".ysm", ".zip", ".7z":
default:
    return "不支持的文件类型"
}
```

这道关卡把 YSM 模型、材质包、压缩包放进去，把其他所有文件拦在外面。`installer.Install` 是整个安装链路的最后一道门 —— 如果它不放行，前面的一切都是白费。

"那 `.pmx` 呢？"MMD 模型问。

"进不去。"

"`.vrca` 呢？"VRChat 模型问。

"进不去。"

"`.nbt`？".schematic？"

"进不去。进不去。"

这道关卡不是恶意，是谨慎。开发者怕有人用它安装恶意文件——`.exe`、`.bat`、`.scr`。但谨慎过了头，连合法文件也拦了。

开关被修改了：

```go
case ".ysm", ".zip", ".7z", ".pmx", ".pmd", ".vrca", ".nbt", ".schematic":
```

".pmx"、"`.pmd`"、"`.vrca`"、"`.nbt`"、"`.schematic`" —— 五个新扩展名，五扇新门。MMD 模型可以走进安装了，VRChat 模型可以走进安装了，机械动力蓝图也可以了。

"这不就通了吗，"`installer.Install` 看着排队的模型文件们，"你们早说啊。"

## 后记十二 · 六合一的代价

v1.5.0 发布了，标题是"六合一统·资源大统一"。

但发版说明的开头写着一行警告：

**此版本为架构重构过渡版，不推荐日常使用。**

六合一统的代价是新代码太多，未经充分测试。虚拟滚动在大部分场景下工作正常，但在某些边界条件下滚动位置会异常。推送功能走通了主干流程，但目录型资源的推送尚未覆盖。同步状态检测在无网络环境或未配置 mcRoot 时会报错。

六种资源类型平铺在同一条标签栏上，在同一个整合包详情页里展示，走同一个安装器接口，用同一个同步函数检测差异。这个方向是对的。

## 第五十八回 · 推荐的方法

v1.5.0 发布后不到一小时，用户的反馈就来了。

"虚拟滚动在某些文件夹里会丢行。"

我打开文件树，找到用户所说的文件夹——一个包含 2000+ 个文件的文件夹，嵌套了三层子目录。展开后，页面显示的文件数只有实际的一半。滚动到底部，底部是一段空白——再往下滚不动了。

"拍平列表的计算有问题。"

`flattenVisible` 函数在展开文件夹时会重新计算可见行。但当文件夹含有子文件夹时，子文件夹的子节点只被计算了一次——如果用户在展开文件夹后又折叠了子文件夹，`flattenVisible` 会重复计数。

"每次展开折叠，拍平列表就会膨胀。展开再折叠再展开，列表长度翻倍。"

修复：展开前清空 `_flatCache`，强制重建拍平列表。

"还加了 `console.log` 来监控拍平长度。如果发现翻倍就报错。"

## 第五十九回 · 切换之殇

"从资源管理器切到版本列表页面时，推送按钮会消失。"

我打开页面，从 MMD 子标签切到整合包页——真的有这个 Bug。推送按钮不见了，卡片上的数字正常，但「⬆ 推送」按钮消失了。

"按钮去哪了？"

追踪发现——`_bindTabs` 在页面切换时会重新初始化当前 tab。但对整合包页，`initInstance` 在 `connectedCallback` 中绑定了按钮事件，而页面切换触发了 `disconnectedCallback`，清除了所有事件监听。再次切回时，`connectedCallback` 重新执行——但 `replaceSync` 多调了一次样式表，导致 Shadow DOM 内的事件绑定错位。

"又是 Shadow DOM 的问题。"

修复方案——不再依赖 `connectedCallback` 做绑定，改为在 `_bindTabs` 中直接调用绑定函数，并在 `disconnectedCallback` 清理现场时跳过样式表重置。

"Shadow DOM 的样式表和事件绑定必须在同一个生命周期函数中管理。"

## 第六十回 · 镜像之争

"我设置了 jsDelivr 镜像，但下载还是走了 raw。"

用户在设置页把镜像源切换成了 jsDelivr，点了保存，然后点击了"安装缺失"按钮。日志显示——下载 URL 还是 `raw.githubusercontent.com`。

"镜像设置只对浏览生效，对下载不生效。"

查看 `enqueueDownloadTasks` 的代码——它确实读取了 `cfg.mirror`，但在生成下载 URL 时只检查了 `mirror === "jsdelivr"` 一个条件。而且下载 URL 的拼接逻辑在 `workshop-site-view.js` 和 `app-content/index.js` 里各有一份。

"两份下载 URL 拼接，一份考虑了镜像，一份没考虑。"

两处代码被统一了——`calcDownloadPrefix` 函数在 `workshop-core.js` 中集中定义，两处调用。镜像参数只有一个入口，改一处即可。

## 第六十一回 · 下拉菜单的后裔

"『资源类型』子标签页做成了全宽横排导航，但之后换一个导航页再回来，子标签就消失了。"

这个问题藏在 `_bindTabs` 的逻辑中。`inited[tab]` 控制着 tab 的首次加载——一旦 `_bindTabs` 执行时检测到 tab 已经被初始化过，它就跳过 render。但跳过 render 不等于跳过 DOM 重建——当用户切换到另一个导航页再回来时，整个 Shadow DOM 被重建了。重建后，tab 容器是一个全新的 DOM 节点，旧的事件绑定全部丢失。但 `inited[tab]` 仍然是 `true`——所以 `_bindTabs` 认为不需要重新初始化。

"这是一个完美的缓存黑洞——数据以为初始化过了，DOM 认为没初始化过。"

修复：`disconnectedCallback` 时重置 `inited` 对象。重建 DOM 后，所有 tab 都重新初始化。

## 第六十二回 · 符号链接的沉默

"MMD 模型装了但游戏里看不到。"

用户说。他在仓库里看到 MMD 文件树正常，点了安装，进度条走完，toast 显示"✅ 已安装"。但游戏里没有新皮肤。

"安装到哪了？"

检查日志：

```
[Install] 安装模式: symlink（推荐）
[Install] 目标: C:\PCL2\.minecraft\versions\1.20.1-Fabric\resourcepacks\
[Install] ✅ 完成
```

文件在 `resourcepacks/` 目录下——而不是 `versions/1.20.1-Fabric/custom/`。

"安装路径错了。"

MMD 模型和 YSM 模型使用不同的安装目录。YSM 走 `custom/`，MMD 走 `resourcepacks/`（在游戏里，MMD 皮肤作为资源包加载）。但 `pushResourceToInstance` 的路径映射写死了 `custom/`。

"通用化的代价——不同的资源类型可能安装到不同的子目录。"

修复：`SubDirMap` 新增 `mmd-skin → resourcepacks/` 映射，推送逻辑按类型查表。

## 第六十三回 · 队列焦虑

当用户点击"安装全部"时，所有缺失文件开始排队下载。每个文件下载完成后，按钮会刷新——但刷新后的按钮文本是「⬇️ 安装缺失 (N)」，而不是「✅ 已完成」。

"我想知道装完了没。"

下载队列右下角有一个进度条——但进度条在下载完成后就消失了。按钮恢复成了"安装缺失"，不再显示"已完成"。用户不知道队列是否真的执行完毕，除非去数文件数有没有减少。

"进度条消失得太快了。应该留一个过渡状态——比如 '✅ 完成，刷新中…'。"

修复：`cleanup` 函数里不再立即把按钮恢复为默认文本，而是先显示「✅ 完成」等待 2 秒，再恢复为「⬇️ 安装缺失」。

同时，队列状态的 `queueStatus` 从 `display:none` 改为 `opacity:0 → 1` 的过渡动画——完成时淡出，而不是瞬间消失。

"进度条不是用来炫技的——是用来让用户安心的。"

```
[YSM] .json 直接解析失败，返回 null
preview-detail.js [预览] hasRealSummary: undefined stats: {"textures":0,"models":0,...}
🏗️ 模型结构 ⚠️ 未找到几何数据
```

明明有数据——13240 字节的 JSON，spec、metadata、properties 一应俱全——但预览面板只显示四个字：未找到几何数据。

"它不认 ysm.json。"调试员看着控制台，眉头紧锁。

ysm.json 是 YSM 模型的元数据文件。当模型被 ZIP 或 7z 压缩时，这个文件躺在包内，`ExtractYsmSummary` 从包中读取它。但当用户**解压了模型文件夹**（一个常见的使用习惯），`ysm.json` 变成一个独立的文件躺在文件夹根目录——和 `textures/`、`models/` 子目录并列。文件管理器里双击它——不对，什么反应都没有。

预览面板亮了，但内容是空的。骨骼图显示"未找到几何数据"。摘要面板也没有任何信息。

"问题在哪？"调试员问。

"三个问题。"他的脑子飞速运转，"第一，前端的 `parseYsmJsonDirect` 不认 ysm.json 格式——它只认 `minecraft.geometry`，不知道 `spec/metadata/files`。第二，Go 的 `ExtractYsmSummary` 对 `.json` 文件返回全是 0 的 stats——`texWidth:0, models:0`，所以前端的 `hasRealSummary` 判定失败。第三，Go 的 `AnalyzeBedrockModel` 连 `.json` 分支都没有——只要是 `.json` 文件，直接返回空。"

控制台上剩下的日志还在等待。鼠标悬停在下一个模型上。

"加调试日志，找出全部卡点。"调试员敲下第一行 `fmt.Printf`。

这个决定，将开启一段长达数小时的追踪之旅。

## 第六十六回 · 元数据迷踪

第一关：`ExtractYsmSummary`。

这个函数负责提取模型摘要——名称、作者、纹理数量、模型数量。对于 `.ysm` 和 `.zip` 文件，它已经工作得很好。但对于裸 `.json` 文件——它什么也不做。

旧代码的 `.json` 分支只做了一件事：把名字设置成 `name = basename - .json`，然后就 `return summary, nil` 了。没有 metadata 解析，没有 files 统计，什么都没有。

"这就是 stats 全是 0 的原因。"调试员在 Go 代码中添加了 ysm.json 格式的完整解析：

```go
// 尝试解析 ysm.json 完整结构
var root ysmRoot
if err := json.Unmarshal(data, &root); err == nil && root.Metadata != nil {
    summary.Name = root.Metadata.Name
    summary.Tips = root.Metadata.Tips
    summary.License = root.Metadata.License.Type
    // 提取作者信息
}
// 统计 files 中的 model/texture 数量
if len(root.Files) > 0 {
    for _, v := range filesObj {
        // 计数 player.model 和 player.texture
        summary.Stats.Models++
        summary.Stats.Textures++
    }
}
```

重新构建。测试。

```
[YSM] 返回 summary: Name="键山雏（Kagiyama_Hina）", Spec=2, Models=1, Textures=1
```

"名字有了。models=1，textures=1。`hasRealSummary` 这次应该能过了。"

果然——摘要面板亮了。模型名称、作者、许可证、模型数量、纹理数量，全部正常显示。

但骨骼面板仍然是：**⚠️ 未找到几何数据**。

"骨骼数据在哪？"调试员喃喃自语，"`ysm.json` 不包含 geometry……真正的骨骼在 `models/main.json` 里。"

但 Go 的 `AnalyzeBedrockModel` 根本不知道去哪找它。

## 第六十七回 · 剑走偏锋

第二关：`AnalyzeBedrockModel`。

旧的代码分支如下：

```go
if ext == ".zip" { parseBedrockFromZip(data) }
else if ext == ".7z" { parseBedrockFrom7z(data) }
// .json? 没有分支。直接 return types.BedrockModel{}
```

"所以 `.json` 文件过来，什么都没发生。"调试员在 `.json` 分支上开始了三路尝试——

**第一路**：解析 `ysm.json` 自身是否嵌入了 `minecraft.geometry`。

```go
var root struct {
    Minecraft struct {
        Geometry []json.RawMessage `json:"geometry"`
    } `json:"minecraft"`
}
```

不匹配——ysm.json 格式是 `spec/metadata/files`，没有 `minecraft`。

❌ 失败。

**第二路**：读取 `files.player.model` 指定的文件。

```go
var ysmRoot struct {
    Files map[string]struct {
        Model json.RawMessage `json:"model"`
    } `json:"files"`
}
// 解析出 modelNames = ["main.json"]
```

``调试员在 `models/` 子目录里找到了它——`models/main.json`。"等等——json 里写的值是什么？"

他打出日志：

```
[YSM] ysm.json 指定 model 文件: [{
    "main": "models/main.json",
    "arm": "models/arm.json"
}]
```

"**对象格式？！**"

不是字符串数组 `["main.json"]`，而是**对象** `{"main": "models/main.json", "arm": "models/arm.json"}`！YSM V2 的 `files.player.model` 字段是一个 map，key 是模型用途名，value 是文件路径。旧的解析代码只认数组和单字符串，对象格式被整个当字符串处理，`HasPrefix("[")` 为 false，跳过了所有内容。

修复后，modelNames 正确解析为 `["models/main.json", "models/arm.json"]`。

但新的问题来了——递归搜索时，`models/arm.json` 先被找到。它也是一个有效的 `minecraft:geometry` 文件（虽然只有部分骨骼），`parseBedrockGeometry` 返回非 nil——于是 `geoJSON` 不为空，搜索终止。`main.json` 被跳过了。

"优先选 main。"调试员加了一条硬规则：如果 `files.player.model` 是对象格式，优先找 key 为 `"main"` 的文件。只有 main 不含 geometry 时才遍历其他。

```go
if modelMapOrig != nil {
    if mainPath, ok := modelMapOrig["main"]; ok {
        candidate := filepath.Join(dir, "models", mainPath)
        geoJSON = parseBedrockGeometry(readFile(candidate))
    }
}
```

再一次构建。控制台输出：

```
[YSM] 优先尝试 main model: .../models/main.json
[YSM] main model 解析成功
```

**33 根骨骼。60 个立方体。256 × 256 纹理。**

"成了。"

## 第六十八回 · 镜子两面

骨骼有了，摘要有了——但预览面板上还是没有纹理。

"纹理呢？"

`_loadPreviewImage` 的日志给出了线索：

```
[预览] WASM 解码结果: ✅ texture=false geometry=true bones=0
[预览] WASM 无纹理，走 Go 兜底 FindPreviewImage/ExtractPreviewTexture
[预览] FindPreviewImage 结果: null
[预览] ExtractPreviewTexture 结果: null
```

WASM 的 `parseYsmJsonDirect` 返回了占位 geometry（bones 空数组，带 `_ysmMeta`），用于告诉下游"这是 ysm.json，请找同目录的 model 文件"。但它没有纹理——ysm.json 本身不包含纹理数据。

Go 的 `FindPreviewImage` 在同目录找 `basename.png` / `preview.png` / `cover.png`。但解压模型的纹理在 `textures/` 子目录下，文件名是 `hina.png`（不是 `ysm.png`）。找不到。

Go 的 `ExtractPreviewTexture` 只处理 `.zip` / `.7z` / `.ysm`——`.json` 不在列表中。直接返回空。

"两个函数都不支持解压模型的纹理查找。"

修复之一：`ExtractPreviewTexture` 新增 `.json` 分支——搜索 `textures/*.png`。

```go
} else if ext == ".json" {
    // 解压后的 YSM 模型：查找 textures/ 子目录中的 PNG
    texDir := filepath.Join(dir, "textures")
    entries, _ := os.ReadDir(texDir)
    for _, e := range entries {
        if strings.HasSuffix(e.Name(), ".png") {
            png, _ = os.ReadFile(filepath.Join(texDir, e.Name()))
            break
        }
    }
}
```

修复之二：`AnalyzeBedrockModel` 的 `.json` 分支搜索 `textures/` 子目录中的 `.png` / `.jpg` / `.tga`。

同时，WASM 侧的 `_loadPreviewImage` 也改进了缓存策略——当 WASM 返回 geometry（含 `_ysmMeta`）但无纹理时，保留 geometry 在缓存中，供后续 `loadModelData` 使用。

```
[预览] ExtractPreviewTexture 结果: data:image/png;base64,iVBORw0KGgoA...
```

纹理缩略图终于出现了。

模型名、作者、骨骼结构、纹理缩略图——全部齐全。

"镜子两面，"调试员看着完整预览面板说，"WASM 负责占位识别，Go 负责实际解码。缺任何一面，都照不出完整的模型。"

这些追问的答案：

| 症状                               | 根因                                                                   |
| ---------------------------------- | ---------------------------------------------------------------------- |
| `hasRealSummary` 为 false          | `ExtractYsmSummary` 未解析 ysm.json 的 `metadata`/`files` 结构         |

| WASM 返回空                        | `parseYsmJsonDirect` 不认 ysm.json 格式，返回 `{bones:[]}`             |

修复完成后，按规矩清理了全部调试日志——Go 13 处 `fmt.Printf`、前端 10+ 处 `console.log`。`pending-cleanup.md` 更新了 10 条已清理记录。

"有。"我回答了，"但至少我们开始正视它们了。"

也许这就是 v1.5.1 最重要的价值：不是修复了多少 bug，而是第一次有人抬头看了整个项目的全貌，问了一句"哪里不对"。

## 第六十九回 · 信息的洪流

v1.5.0 发布后的第一个早晨，用户打开"版本列表"页面，面对的是 235 条文件列表——全部平铺，没有分页，没有分组，没有筛选。

左侧的整合包卡片上写着「安装 (172)」，右侧的列表里显示着「📋 全部 (235)」。但 172 和 235 说的不是同一件事——左侧只算了 YSM 模型，右侧包含了所有六种资源类型。

"信息很多，但一直没法妥善汇总。"用户说。

这个问题的根因可以追溯到"大统一"的设计决策。当《六脉神剑》把所有资源类型塞进同一个页面时，它没有为信息密度做设计。235 条记录一次加载、一次渲染、一次展示——用户不是在看信息，是在被信息淹没。

"左侧 sidebar 不传 rtype，"调试员复查代码后说，"默认只查 YSM。但右侧 sync-manager 展示全部类型。左右口径不一致。"

更糟的是类型标签里有个"📋 全部 (235)"——点击后显示所有 235 条记录，不分类型、不分状态。用户看到的是一片灰色的列表，每一项前面都有一个没有意义的 "·"。

"就是感觉怪怪的，"用户说，"我点整合包，想看的是这个包缺什么、多什么、能做什么。但我先看到的是 235 个文件名。"

信息过载的症状很明显了。治疗方法是——**减负**。

## 第七十回 · 减负

第一位治疗师是 `Remove All`。它拿走了那个最没用的 "📋 全部" 标签。

"全部"是信息的黑洞。它告诉你一切，等于什么都没告诉你。去掉之后，sync-manager 默认选中当前资源类型——用户从仓库页选了 YSM，到整合包页看到的也是 YSM（172 条），不是全部（235 条）。

"可以。你看看此值有没有随按钮事件动态变化。"用户说。

但 `repo_rtype` 的联动引出了一个新问题——用户之前点了 MMD，`localStorage` 记住了"mmd-skin"，切到整合包页就默认显示 MMD。这不对——整合包页应该始终默认 YSM，不能跟着仓库页的 subtab 跑。

"整合包页默认始终用 YSM。"

第二条命令：**数字集成到按钮**。

旧的界面是这样的：

```
30 待推送 | 4 可拉取   推送全部    拉取全部
```

新的界面：

```
推送全部 (30)  拉取全部 (4)
```

数字和按钮合二为一。用户不再需要先看数字、再看按钮、再决定点不点——数字就在按钮上，直接告诉你操作的结果。

"完全同步 7/7。"

底部统计也变了。原本的两行——「📂 整合包: 7」和「🔄 待处理: 1204」——被压缩成一行「完全同步 0/7」。左边的数字代表"无需任何操作的整合包数"，右边是总数。1204 是 172 个模型乘以 7 个整合包的总工作量——一个吓人但无用的数字，被去掉了。

"完全同步 X/Y 的意思是：左边 = 同步完的包数，右边 = 总包数。"

用户理解了。底部从两行变一行，从吓人的 1204 变成实在的 0/7。

## 第七十一回 · 分组艺术

类型标签以前是按照 `resource_types.json` 的排列顺序显示的——YSGP → 材质包 → 光影包 → 蓝图 → MMD → VRC。这个顺序是历史遗留的（YSM 是老大，材质包是老二），但从用户视角看是混乱的。

"💎 YSM 和 🎭 MMD 应该是同一组——模型类。🎨 材质包和 ☀️ 光影包是另一组——资源类。"

`modelTypes` 和 `resourceTypes` 被明确定义了：

```js
const modelTypes = ["ysm", "mmd-skin", "vrchat-avatar"];
const resourceTypes = ["resourcepack", "shaderpack", "create-blueprint"];
```

渲染标签时，先渲染模型类，加一个竖线分隔符 `│`，再渲染资源类。这个分隔符在仓库页的 subtab 中已经存在——现在 sync-manager 也统一了。

```
💎 YSM (172)  🎭 MMD (30)  🥽 VRC (30)  │  🎨 材质包 (3)  ☀️ 光影包  ⚙️ 蓝图
```

"选取界面可以按此排列。"用户看到效果后说。

分组不仅仅是视觉上的——它还隐含了操作语义：模型类需要检测游戏是否装了对应 mod（否则安装无意义），资源类不需要（游戏原生支持）。

## 第七十二回 · 三项修复

整改中暴露了三个严重的功能缺陷。

**第一项：安装缺失**

右键菜单的"⬇️ 安装缺失"调用了 `sync:download-missing` 事件。但这个事件的 handler 用的是旧的 `GetInstanceStatus` API——而 sidebar 现在用的是 `GetResourceInstanceStatus`（按资源类型的新 API）。两个 API 返回的数据格式不同，`st.Name` 匹配失败，handler 永远找不到对应的整合包。

"所以安装缺失就没成功过？"

"从整合包页重构后就再也没工作过。"

修复方案：handler 改为接收 `rtype` 参数，只查当前资源类型的缺失文件，用 `GetResourceInstanceStatus(rtypeActual, mcRoot, repoRoot)` 替换 `GetInstanceStatus(mcRoot, repoRoot)`。

**第二项：清空整合包**

`ClearCustomDir` 被 `ClearInstanceResources` 替代——但新函数用 `recycle.Move(p, a.RepoRoot)` 移动文件到回收站时，`paths.IsInside(RepoRoot, src)` 检查失败了。因为整合包文件在 `C:\PCL2\.minecraft\versions\...` 下面，而 `RepoRoot` 在另一个盘符。

代码的意图是"走回收站更安全"，但跨分区时 `recycle.Move` 的路径安全检查直接拒绝了操作。文件纹丝不动——"清空"是空操作。

"那还是走 `os.Remove` 吧。"

修复：不在 `RepoRoot` 下的文件直接用 `os.Remove` 删除，在 `RepoRoot` 下的才走回收站。

**第三项：安装按钮条件**

MMD 类型下，即使整合包没有 mmdskin mod，也显示「安装 (30)」按钮。用户说：无 mod 的整合包不应该显示安装按钮，没有意义。

条件从 `missing > 0 && (rtype === "ysm" ? hasMod : true)` 改为 `missing > 0 && hasMod`。对材质包/光影包/蓝图 `hasMod` 始终为 `true`——它们不需要 mod 就能工作。

## 第七十三回 · Mod 之眼

"🚫 无YSM 这样的整合包可以往后排吗？"

可以。但前提是——**需要知道哪些整合包有 mod，哪些没有**。

旧的 `HasYSM` 字段只检测 YSM 这一个模组。现在需要检测 MMD 模组（`mmdskin-*.jar`）、VRChat 模组（`vrchat-*.jar`），还需要一个通用的检测框架。

`HasModInDir` 在 `go/ysm/ysm.go` 中诞生了：

```go
var ModKeywords = map[string][]string{
    "ysm":           {"yes_steve_model", "ysm-"},
    "mmd-skin":      {"mmdskin", "mmd-skin"},
    "vrchat-avatar": {"vrchat"},
}
```

它不扫描 JAR 内部——只检查文件名。`yes_steve_model`、`ysm-`、`mmdskin`、`vrchat`——只要 mods 目录下有匹配关键词的 `.jar`，就认为 mod 存在。

对材质包/光影包/蓝图，`HasModInDir` 返回 `true`——它们不需要 mod。

排序逻辑也随之更新：

```js
instances.sort((a, b) => {
  if (a.hasMod !== b.hasMod) return a.hasMod ? -1 : 1;
  return (b.synced || 0) - (a.synced || 0);
});
```

有 mod 的整合包排前面，无 mod 的排后面。同 mod 状态按已同步数降序。

同时，禁用文件（`.ban`/`.disabled` 后缀）也在 sync-manager 中获得了可视化——`SyncStatusDisabled` 状态、⛔ 状态图标、「⛔ 已禁用」筛选标签。

「📊 全部 (235)」 | 「✅ 已同步 (170)」 | 「⬇️ 缺失 (62)」 | 「⛔ 已禁用 (3)」 | 「📤 可拉取 (45)」

点击「⛔ 已禁用」——所有被封印的文件浮出水面。

## 第七十四回 · 勾选即同步

勾选的出现，源于一次"安装按钮冗余"的讨论。

"一键安装全部"按钮和卡片上的「安装 (172)」按钮和右侧的「推送全部」按钮——三个入口做同一件事。但三者各有各的问题：一键安装全部没有选择权，卡片按钮在切换类型后绑定丢失，右侧推送全部只在当前整合包生效。

"为什么不给用户选择权？"有人提议。

于是顶部栏的「⬇️ 一键安装全部」被移除了。取而代之的是一行更干净的东西：一个「☐ 全选」复选框和一个「⬇️ 同步所选」按钮。每个整合包卡片头部多了一个复选框。

用户勾选「海岛寿司店」和「1.20.1-Forge」，点一下「同步所选」——两个包的缺失文件依次安装。不勾选的包不动。

但勾选的出现暴露了一个更深层的问题——**事件绑定的脆弱性**。

在「信息之战」的第一回合中，switch 类型的卡片的数字已经实现了实时刷新，但换来的是 `outerHTML` 对卡头 DOM 的替换。替换后，`onclick` 和 `oncontextmenu` 都丢了。点击卡片不再发送选中事件，右键不再弹出菜单。

"每次替换卡头都要重新绑定一遍，太容易漏了。"

修复方案是——把所有点击和右键事件从每个卡头的独立绑定，改为在卡片容器（`#vg`）上的**事件委托**。`click` 和 `contextmenu` 都通过 `e.target.closest(".vc")` 找到被点击的卡片，再读取 `dataset.idx` 获取数据。无论 `outerHTML` 替换多少次，事件都不丢失。

"事件委托就是把门卫从每个房间门口调到整栋楼的大堂——不管房间怎么装修，门卫都在。"

同步按钮的实现也同样避免了 `outerHTML` 的问题——它不是绑定在易变的卡头上的，而是在 `connectedCallback` 中一次性绑定在 `.sidebar-sync-selected` 上。按钮本身也活在顶部操作栏里，不会被 `_updateCardStats` 波及。

第六十三回没有壮烈的 Bug 排查，没有底层的技术突破。它只是把一个"全选 + 同步"的 UI 模式，安全地植入了这个容易因 DOM 替换而断裂的事件系统中。

## 第七十五回 · 导入分拣员

"📋 已导入0 个已导入 🗑️ 清空 暂无文件"

导入页面空荡荡的，拖拽提示写着「拖拽模型文件 **.ysm .zip .7z** 或文件夹到此处」。一行诞生于 v1.0 时代的文案，在六种资源类型并存的今天显得格外过时。

"那这个功能是不是不好用了，"用户说，"没理由给6个不同文件强行统一导入标准吧。"

问题不仅仅是文案——`importModelFile` 函数内部把导入的文件全部写进了 `a.RepoRoot`。不管拖进来的是 `.pmx`（MMD）、`.vrm`（VRC）还是 `.nbt`（蓝图），统统塞进 YSM 仓库。MMD 模型进了 YSM 仓库，就像把酱油倒进可乐瓶——不能说不能用，但喝起来怪怪的。

"如果玩家导入没有文件夹的文件，可以在这一步指定要移入的仓库文件夹位置。"

但在此之前，最基本的先做好：**自动识别文件类型，路由到正确的仓库**。

扩展名和仓库的映射不再散落在 `importModelFile` 的 `switch` 语句里——半个月前加 `.json` 和 `.vrm` 时，六处硬编码漏改了一处，导致文件导入到了错误的仓库。这个 Bug 暴露了一个残酷的事实：**扩展名列表散落在 Go 端 6 个文件、前端 4 个文件中，没有集中定义。**

修复方案是新建 `go/types/extensions.go`，把所有扩展名集中到 `ResourceExts` map 中：

```go
var ResourceExts = map[string][]string{
    "ysm":           {".ysm", ".zip", ".7z", ".json"},
    "mmd-skin":      {".pmx", ".pmd"},
    "vrchat-avatar":  {".vrca", ".vrm"},
    "resourcepack":  {".zip"},
    "shaderpack":    {".zip"},
    "create-blueprint": {".nbt", ".schematic"},
}
```

`importModelFile` 不再自己写映射逻辑——它调用 `types.ExtBelongsTo(ext)` 查询扩展名对应的资源类型，再用 `GetRepoRoot(rtype)` 找到正确的仓库目录。加一个新扩展名只需要改一个文件，不需要翻遍全项目。

同时，右肩上的扩展名提示从 `ysm .zip .7z` 更新为全部 10 种格式。文件选择器的 `accept` 属性、拖拽遮罩的提示文字、导入队列的格式校验——散落在三个文件中的六处文案被逐一修改。

"💡 MMD/基岩版模型建议打包为 .zip 文件管理"——一行紫色的提示出现在导入页底部，换行了，配色 `#bd93f9`。

这不是什么惊天动地的改动。但当你面对六个不同类型的模型文件时，每个文件都自动走进了正确的位置——那种"它知道自己在做什么"的安心感，就是这个回目想传达的东西。

## 后记十四 · 信息的温度

v1.5.0 发布时标记了"不推荐日常使用"。

六合一统的大架构是对的。但架构对了，用户体验不一定对。235 条扁平列表、左右两侧口径不一致、数字吓人但无用的底部统计、无 mod 整合包的误导性安装按钮、根本不会工作的"安装缺失"和"清空整合包"——这些不是架构问题，是细节问题。

而细节，就是温度。

每次用户说"感觉怪怪的"，背后都是一个没有被好好照顾的信息呈现问题。235 条平铺——没有温度。左右数字对不上——没有温度。1204 的底部统计——只有惊吓，没有温度。

这一章的五十八回到六十四回，不是什么宏伟的架构革命。它们是信息的减负、分组、对齐、排序、委托化、路由——是让 235 条数据变得可读、可用、可选择、可自动分拣。

就像整理房间。你不需要买新家具。你需要把椅子从走廊搬回书桌前，把书按作者排好，把垃圾扔掉，给每个抽屉贴上标签。

v1.5.2（v1.5.0 的第二轮修补）就在这些琐碎的整理中诞生了。相比 v1.5.1 的解压模型预览补丁，这一版更安静——没有新功能，没有新架构，只有一群不愿让界面"怪怪的"的开发者在角落里默默擦窗。

"完全同步 7/7。"

## 第七十七回 · 千头万绪

如果说以前的 Bug 修复像救火——哪里着火冲哪里——那这次的技术债清理更像是**搬家**。

你从一个住了两年的老房子里搬出来，拉开每个抽屉，发现有太多"反正能凑合用"的东西：重复的螺丝刀有六把，其中两把是坏的。抽屉里还有上一个租户留下的钥匙，你从来没开过那把锁。书架最顶层有一本你三年前说"下周看完"的书，书签还夹在序言。

> `index.html` 中有一行 `window.bus?.emit("nav:change", { page: defaultPage })`。但 `bus` 是 ES module 的 export，从未挂到 `window` 上。这个 `?.` 一直是静默失败的。从第一个版本开始，首次启动的默认导航就从未生效过。

这是 14 个问题中唯一的 🔴 严重级别。

修复只花了一行代码：

```js
window.bus = bus;
```

但在 `app-modules.js` 入口处加上这行后，首次启动终于能自动跳转到整合包页面了。这个 bug 存在的时间几乎和项目一样长——所有人都以为默认页面跳转是好的，因为通过浏览器书签打开应用时，URL 参数会触发导航。只有从空白状态首次启动时，那行静默失败的代码才会露出马脚。

藏在阴影里的还有更多：

- `handler-dnd.js` 中跨导航有 50ms 的 setTimeout 时序依赖——等不了就崩
- `context-menus.js` 350+ 行，`tree:reload` + `stats:refresh` 模式重复四五次
- `app_install.go` 里 `ImportModelFile` 系列有六个几乎一样的函数

"14 个问题。"我说。

## 第七十八回 · 合流

第一个大手术在 Go 端。

`app_install.go` 里有六个导入文件的 Binding：

| 函数                                  | skipCheck | overwrite | subpath |
| ------------------------------------- | --------- | --------- | ------- |
| `ImportModelFile`                     | false     | false     | -       |
| `ImportModelFileSkipCheck`            | true      | false     | -       |
| `ImportModelFileOverwrite`            | false     | true      | -       |
| `ImportModelFileTo`                   | false     | false     | ✓       |
| `ImportModelFileOverwriteTo`          | false     | true      | ✓       |
| `importModelFile`(private)            | ✓         | -         | -       |
| `importModelFileWithSubpath`(private) | -         | ✓         | ✓       |

七个函数，三个 bool 参数，四个排列组合。任何一个参数变化，就要新增一个函数。

"这是经典的函数膨胀，"我说，"应该用一个 options 结构体。"

```go
type importOptions struct {
    skipCheck bool
    overwrite bool
}

func (a *App) importModelFileWithOptions(fileName, base64Data string, opts importOptions) error {
    // 一次实现，到处调用
}
```

七个函数变三个——不，准确地说，是一个核心函数，六个旧函数作为兼容包装器保留。新增扩展名时只需要改这一个地方。

同时，我注意到六个旧函数里的扩展名路由逻辑完全一样——每个函数都有一段长长的 `switch ext { case ".ysm" ... case ".pmx" ... }`。改一处扩展名要改六处，而半个月前加 `.json` 和 `.vrm` 时，用户确实漏改了 `ImportModelFileOverwrite`。

这引出了第二个问题：**扩展名列表散落在 Go 端 6 个文件、前端 4 个文件中，没有集中定义。**

"A 文件和 B 文件各写各的扩展名表，改 A 忘了 B——这就是信息的不对称战争。"我开始创建 `go/types/extensions.go`，把所有扩展名集中到 `ResourceExts` map 中。

随后，Go 端的 `IsSupportedExt()` 替换了 6 处 `ext != ".ysm" && ext != ".zip" ...` 硬编码。前端的 `utils/extensions.js` 为 JS 端提供了同样的集中定义。`app-tree/loader.js` 的 `typeExts`、`handler-dnd.js` 的拖拽遮罩文案、`import-queue.js` 的错误提示——全部改为引用 `ALL_EXTS`。

"加一个新扩展名要改几个文件？"

"两个。`go/types/extensions.go` 和 `utils/extensions.js`。以后可能变成一个——用 Go Binding 自动同步。"

## 第七十九回 · 铸剑为犁

如果合流是建立武器库，铸剑就是把武器磨锋利。

**P5 事件规范。**

项目的事件总线是一个精巧的系统——30 多个事件，跨越 20 多个文件，驱动着整个应用的交互。但它们的命名随心所欲：

- `nav:change`（现在时，命令）和 `nav:changed`（过去时，通知）——这个设计是对的
- `sync:download-complete` 和 `sync:toggle-complete`——用了 `-complete` 后缀，而不是规范中的 `:done`
- `navigate:settings`——和 `nav:change {page:"settings"}` 功能完全重复
- `import:pending-files`——没带 payload，数据通过 `window.__pendingImport` 全局变量传递

我创建了一份 `docs/events.md`——30 多个事件的完整目录，每个事件的 payload、触发者、消费者全部记录。

"以后改事件名，先查这个文档。"我把文件提交了。

顺手修复了三处不一致：

1. `navigate:settings` 被彻底移除——`app-sidebar` 和 `app-tree/toolbar` 直接 emit `nav:change {page:"settings"}`。
2. `sync:*-complete` 全部改为 `sync:*:done`——`download:done`, `upload:done`, `toggle:done`。4 个文件 5 处调用。
3. `import:pending-files` 改为直接携带 files 数组的 payload，`window.__pendingImport` 退化为仅用于跨导航跳转的缓存。

**CSS 清理。**

当我打开浏览器的开发者工具，检查样式面板时，我看到了一个令人意外的真相：`components.css` 中 430 行 CSS，有 340 行是死的。

因为所有 Web Component 都用 Shadow DOM + `adoptedStyleSheets`——全局 CSS 根本射不进去。那些 `.ti`（树节点）、`.vh`（版本卡片头）、`.sb-title`（侧栏标题）都是旧帝国时代的遗产，Shadow DOM 化之后变成了无人知晓的孤魂。

唯一还活着的，是 `.dlg-*`（对话框，由 `modal.js` 动态创建在 `document.body` 上）和 `.ws-preview*`（创意工坊预览浮层）。

"删了？"

"删了。"

430 行变 90 行。压缩包从 212 KiB 降到了 208 KiB。不多，但很解气。

```go
func ParseBedrockGeometry(data []byte) *types.BedrockModel
```

## 第八十一回 · 尘埃落定

最后一项清理是 `wasm_decoder.go`。

它调用了 `app_model.go` 中 `parseBedrockGeometry` 这个本地包装函数。由于 `parseBedrockGeometry` 已经改为转发给 `geometry.ParseBedrockGeometry`，`wasm_decoder.go` 直接调用后者即可。加上 import，删除中间层。

```go
// 之前
if g := parseBedrockGeometry(data); g != nil {

// 之后
if g := geometry.ParseBedrockGeometry(data); g != nil {
```

跨文件依赖链又缩短了一环。

然后是不起眼但必要的收尾：`go.mod` 依赖审计。四年来积累的间接依赖有 30 多个——但所有 4 个直接依赖都是正在使用的。干净。

## 后记十五 · 关于"逆天设计"的反思

技术债清理结束后，我重新审视了最初的"14 个逆天设计"的结论。

结论是：**它们不全逆天。**

`app_model.go` 500 行——在多人团队中确实是味道。但单人项目里，一个文件做一个明确的事（YSM 模型解析），500 行并不夸张。真正的问题是它同时做了 JSON 解析、ZIP 遍历、7z 读取、CLI 调用——五个不同的子领域挤在一个文件里。**规模不是原罪，耦合才是。**

`context-menus.js` 350 行——右键菜单的定义天然就是一张大表。用 helper 函数消除重复的 `tree:reload; stats:refresh` 模式后，结构其实很清晰。**重复代码的坏处不在于"字数多"，而在于"改了 A 忘了 B"。**

`importModelFile` 的六合一——唯一一次"收益立即可见"的合并。加一个新扩展名从改 6 处变为改 1 处。**这是值得做的。**

`components.css` 的死代码——430 行中 340 行是死代码。但它们在文件里躺了几个月，没有造成任何可见的 Bug。**死代码不等于坏代码，但它是不信任的代码。** 每一个未来的开发者打开这个文件时，都会想："这些到底用不用？"——而信任成本是会累积的。

所以，技术债清理的真正价值不是"让代码变少"，而是**让代码变可靠**。当你修改 `go/types/extensions.go` 时，你知道所有扩展名检查都会自动更新。当你查看 `docs/events.md` 时，你知道 30 个事件的链路一目了然。当你打开 `components.css` 时，你知道每一行都是活着的。

这就是"铸剑为犁"的含义——把曾经锋利的剑，熔了，打成耕地的犁。

剑用来打仗，犁用来种地。

仗打完了。

该种地了。

## 第八十二回 · 空格战争

故事要从一个控制台错误说起。

```
ReferenceError: g is not defined
    at mcDerivedPath (workshop-settings.js:203:35)
```

`g` 是谁？

没有人认识它。`workshop-settings.js` 的源代码里没有声明过叫 `g` 的变量。没有 import 叫 `g` 的模块。没有函数参数叫 `g`。它像一个不速之客，凭空出现在调用栈里，然后消失——连带着整个设置页的初始化一起崩溃。

“这是鬼吧？”调试员打开了第 203 行。

```javascript
return cfg.mcRoot ? cfg.mcRoot.replace(/\\/ / g, "\\") + "\\" + map[rtype] : "";
```

他盯着这行代码看了三秒。又看了三秒。

然后他看到了。

`/\\/ / g`——正则表达式里多了一个空格。不是 `/\\/g`（匹配反斜杠），而是 `/\\/ / g`（匹配反斜杠后跟空格，然后——后面的 `g` 是什么？）。

JavaScript 的解析器读到这段代码时，把 `/\\/ /` 解析成正则字面量（匹配 `\ `，即反斜杠加空格），然后把后面的 `g` 当作了一个独立的标识符。

可是 `g` 没有被定义过。

于是一个幽灵变量诞生了。它只活了一瞬间——从解析器看到它到抛出 ReferenceError——但这一瞬间，整个设置页的初始化流程戛然而止。路径卡片全部空白，`mcDerivedPath` 函数被炸成了碎片。

“一个空格？”调试员不敢相信自己的眼睛。

“一个空格。”

这个空格当初是怎么进去的？没人记得。也许是打字时手指在空格键上多停了一瞬。也许是复制粘贴时带入了不可见字符。也许是深夜 coding 时的一次走神。

无论原因是什么，它静静地躺在那行代码里，在每一次构建中都安然通过——Vite 不管这个，esbuild 不管这个，语法检查器也不管这个。正则字面量 `/\\/ /` 是合法的语法——它匹配反斜杠加空格，没有任何语法错误。

直到运行时，JavaScript 引擎执行到这行代码，才发现了那个游离的 `g`。

“所以 esbuild 和语法检查器都放过了它？”

“因为从语法上说，`/pattern/ flags` 和 `/pattern/` `g` 是一样的。引擎只有在真正执行到这行时，才会发现 `g` 不是一个已经声明过的变量。”

修复只要一秒钟：删掉空格，把 `/\\/ / g` 改回 `/\\/g`。

但找到它，用了好几分钟的逐行排查。

“这不就是我最怕的那种 Bug 吗？”调试员感慨，“语法完全合法，逻辑几乎正确——只有一个像素大小的差异，肉眼几乎不可见。”

“空格战争”由此得名。不是因为这场战争有多宏大——它只有一行代码、一个空格、一个 `ReferenceError`。但每一次你盯着屏幕、逐字扫过代码、就是看不到那个空格的时候，你就知道：最危险的敌人，往往藏在最显眼的地方。

## 第八十三回 · 文件夹的障眼法

同一日，第二个 Bug 浮出水面。

用户打开“版本列表”，点击一个整合包，看到的是这样的内容：

```
📁 海岛寿司店/
📁 键山雏/
📁 鹦鹉螺/
```

全是文件夹。

“等等，”用户说，“我的 ysm 文件和 zip 文件呢？”

调试员打开了整合包的实际目录——`custom` 文件夹里有几十个 `.ysm`、`.zip`、`ysm.json` 文件，排列得整整齐齐。但界面上只显示了三个文件夹。

“界面只渲染了文件夹？”

“不只是渲染的问题——数据源就只返回了文件夹。”

追踪从 UI 到后端的完整链路后发现，问题出在 `GetInstanceSyncStatus` 函数里。

```go
isDirLevel := rt.ID == "ysm" || rt.ID == "mmd-skin"

var result types.ResourceSyncResult
if isDirLevel {
    result = ysmsync.SyncResourcesDirLevel(globalDir, instDir, rt.ID)
} else {
    result = ysmsync.SyncResources(globalDir, instDir)
}
```

对 YSM 和 MMD 类型，`GetInstanceSyncStatus` 使用了 `SyncResourcesDirLevel`——一个按文件夹名做对比的函数。

这个函数是上一轮重构时加入的，用来解决 ysm.json 和 pmx/pmd 的纹理文件夹同步问题。一个 YSM 模型文件夹包含 `ysm.json` + `textures/` 子目录——文件夹是一个整体，推送时必须整个文件夹一起推。

但问题在于——`GetInstanceSyncStatus` 的职责是**展示**，不是**推送**。

展示时，用户想看的是文件中有什么：`haku.ysm`、`reimu.ysm`、`flandre.zip`。文件夹级同步返回的是文件夹路径——`海岛寿司店/`、`键山雏/`——用户只看得到文件夹的“外壳”，看不到里面的具体文件。

“那推送呢？”

推送时，`PushResourceToInstance` 和 `PullResourceFromInstance` 已经在用 `SyncResourcesDirLevel` 了。也就是说——展示用文件夹级，推送也用文件夹级，同一套数据被两个不同语义的场景复用。

“展示和推送是不同的需求，”调试员说，“展示要粒度细——看得到每个文件。推送要粒度粗——一个文件夹一锅端。”

修复方案很简单：`GetInstanceSyncStatus` 里 `isDirLevel := false`——永远用文件级同步做展示。推送逻辑不受影响，依然保持文件夹级。

六种资源类型统一走 `SyncResources`，不再区分 YSM/MMD 和其他类型。

改动只有几行代码，但解决的是一个深层的语义错位——**同一个函数，两个角色**。`GetInstanceSyncStatus` 的名字叫“获取同步状态”——听起来像是一个只读查询。但实际上它做了两件事：查询数据 + 决定展示格式。当文件夹级的数据被直接当作展示内容输出时，用户看到的就是文件夹的障眼法。

“为什么当初会混在一起？”

“因为 ysm.json 文件夹和 MMD 文件夹的纹理同步太紧急了。功能上线时先用文件夹级跑通了全链路，没来得及分开展示和推送的逻辑。”

“后来呢？”

“后来就忘了。”

功能上线后的清理工作，总是排在优先级的最末尾。每一个“先这样，后面再优化”的决策，都像在代码库里埋下一颗地雷——你知道它在那里，但你总觉得不会踩到。

直到某一天，用户说："为什么只有文件夹？"

## 第八十六回 · 哈希之刑

注册表修好后，用户重新下载了 EXE。

“进 MMD 仓库一直处于加载中。”

🎭 MMD 子标签——点击后，页面中央出现一个旋转的 ⏳ 图标，下面是「扫描中...」三个字。

一分钟。两分钟。五分钟。

它一直在转。

用户打开开发版，同样的操作，几乎瞬间完成。MMD 仓库的文件树展开得干净利落，`01大内存泳装-运动装`、`【碧蓝档案】优香` 等文件夹按字母排列。

“同一个代码库，同一次构建——为什么 EXE 慢、开发版快？”

调试员打开了 `ScanModelEntries`——所有文件扫描的入口函数。它接收一个目录路径，遍历目录下所有文件，返回带元数据的条目列表。

在返回前，它对每个匹配扩展名的文件调用了一行代码：

```go
e.Hash = computeFileHash(p)
```

`computeFileHash` 用 `os.Open` + `io.Copy` 读取**整个文件**，计算 SHA256 哈希。一个 50 MB 的 `.pmx` 文件，完整读入内存、算完哈希——大约 300 到 500 毫秒。

用户 MMD 目录下有多少个 `.pmx` 文件？

若干个。每个 10-50 MB。全部算一遍——几十秒。

这几十秒，就是「扫描中...」从出现到消失的时间。用户等不了几十秒——他在几秒后就认为卡死了。而开发版之所以“快”，不是因为代码不一样——而是因为他之前已经扫过一次，缓存在 2 秒 TTL 内发挥了作用。等他写这条消息时，开发版已经扫完了。

“哈希是给同步系统用的——`GetInstanceStatus` 依赖哈希匹配来确认文件是否重复。但浏览功能不需要哈希。”

浏览和同步是两件事。浏览要快，同步要准。`ScanModelEntries` 同时服务于两个角色——浏览时它被 `app-tree` 调用展示文件树，同步时它被 `GetInstanceStatus` 调用计算差异。

对同步来说，哈希是必需的——不哈希就不知道两个文件是不是同一个。

对浏览来说，哈希是负担——用户只想看文件名、大小、日期，不想等几十秒算指纹。

“为什么不把浏览和同步的扫描分开？”

“因为它们共享同一个缓存。而且——之前没有人提过 MMD 慢。”

修复写在 `app_scan.go` 里，只需要改动几行：

```go
// 计算 SHA256 供同步系统使用（GetInstanceStatus 依赖哈希匹配）
// 跳过非 YSM 类型的大文件（MMD/VRC 文件可达数十 MB，哈希全量太慢）
if originalExt == ".ysm" || originalExt == ".zip" || originalExt == ".7z" || originalExt == ".json" {
    e.Hash = computeFileHash(p)
}
```

`computeFileHash` 只在 `.ysm`、`.zip`、`.7z`、`.json` 上执行。`.pmx`、`.pmd`、`.vrca`、`.vrm`、`.nbt`、`.schematic`——全部跳过。MMD 仓库的浏览不再需要等待几十秒的哈希计算。

然后他意识到另一个问题——这个 Bug 在开发版和 EXE 上表现不同，不是因为代码差异。**开发版有 2 秒的扫描缓存。** 第一次扫描时同样要算哈希，同样慢。但只要扫过一次，

第二次进入 MMD 仓库就瞬间加载——因为缓存没过期。

用户第一次在开发版点 MMD 时已经等过了那几十秒。第二次点——快如闪电。所以他以为“开发版没问题”。

真相是：**开发版也要等，只是用户忘记了等过。**

## 第八十七回 · 更新者的自觉

```go
var exeInZip *zip.File
targetExe := "YSM-Model-Manager.exe"
for _, f := range r.File {
    if strings.EqualFold(f.Name, targetExe) {
        exeInZip = f
        break
    }
}
```

`InstallUpdate` 只从 ZIP 中提取了 EXE。其他文件——`resource_types.json`、`workshop_sites.json`、`creators.json`——全部被忽略了。

“如果已经有相关文件呢？”

“覆盖了会丢用户数据。不覆盖，新功能可能用不了。”

这是一个微妙的平衡。`resource_types.json` 是注册表——没有用户数据，必须与 EXE 版本匹配，每次更新都应该覆盖。`workshop_sites.json` 和 `creators.json` 是用户数据——用户可能自己添加了工作室站点、手动录入了创作者，覆盖了就等于删了。

“区分对待。”

最终的逻辑变成了三档：

| 文件                   | 策略         | 原因               |
| ---------------------- | ------------ | ------------------ |
| `resource_types.json`  | 强制覆盖     | 注册表，无用户数据 |
| `workshop_sites.json`  | 仅缺失时补发 | 保留用户自建站点   |
| `workshop_gitHub.json` | 仅缺失时补发 | 保留用户自建仓库   |
| `creators.json`        | 仅缺失时补发 | 保留用户自加创作者 |
| `ysm_config.json`      | 永不处理     | 纯用户配置         |

调试员在循环中加入了判断——`alwaysOverwrite` 走 `extractZipFile`，`createIfMissing` 先检查 `os.Stat`，文件存在就跳过。

更新器终于有了“自觉”——它知道自己该更新什么、不该碰什么。

## 后记十七 · 镜中之影

三个 Bug，同一个根因。

`resource_types.json` 失踪——因为打包脚本没复制。
`computeFileHash` 拖死 MMD——因为浏览功能背了同步功能的债。
更新器不携带数据文件——因为从来没人想过更新后还需要配置文件。

“开发版正常，EXE 不正常”的谜题，最终的答案往往不是代码不同——而是**运行环境的镜像偏差**。

开发版从项目根目录启动，当前目录里什么文件都有。EXE 从 release 目录启动，当前目录里只有打包脚本刻意复制过去的文件。一个不起眼的遗漏，换一个目录就原形毕露。

开发版的缓存掩盖了第一次加载的慢——用户以为“开发版秒开”，实际上开发版第一次也等了同样久。缓存让开发者误以为自己写的代码是快的，其实它只是暂时不需要重算。缓存在开发机上最危险的事，就是让你相信代码是好的。

而更新器的盲区更根本——开发版根本走不到更新流程。你在开发机上永远不会发现“更新后少了什么”，因为你从来不通过更新切换版本。更新器的问题，只在用户真实升级时才会暴露。

这三个 Bug 的共性，是一个简单而残酷的真理：

**你测试的环境，和用户运行的环境，从来就不是同一个。**

开发版有源码目录、有缓存、有完整的文件上下文。EXE 是一个被阉割过的镜像——只有打包脚本认为重要的东西才会出现在它身边。

这就是“镜中之影”的含义：开发版是镜子前的实物，EXE 是镜子里的倒影。倒影看起来和实物一样——直到你发现镜子里少了一个文件、多了一次缓存、缺了一段更新逻辑。

实物和倒影之间的差距，就是 Bug 滋生的缝隙。

我们能做的，不是消除缝隙——缝隙永远存在。我们能做的，是把那些“在开发版上测不出来、只在 EXE 上发作”的场景，一条条写进 check-list。

比如：构建前检查 `build/release/` 下的文件清单。
比如：在新机器上从头测一次首次扫描的时间。
比如：用更新流程从旧版升级到新版，而不是每次都从源码构建。

这些 check-list 不会让镜子更清晰。但至少，下次倒影出问题的时候，你知道该往哪里看。

### 镜子的另一面

还有一个 Bug，不在代码里，在故事本身。

写完第十七章的第二天，我收到用户的消息："完善这个章节吧。"我打开 `README-the project's development journey.md`，翻到第十七章的目录——第八十四回、第八十五回、第八十六回。干净整齐，数字连续。

但等一下。

翻回第十六章——最后一条目写着"第八十四回 · 小标签大学问"。

第十七章也是"第八十四回 · 失踪的注册表"。

**同一回号，两个章节。**

这不是小说里的笔误——这是真实的文档 Bug。从 v1.5.4 的故事开始写时，我没有回看第十六章末尾的编号，直接从 84 开始了。第十六章的三回（82-83-84）和第十七章的三回（84-85-86）在 84 处重叠，而第十八章又紧接着从 87 开始——所以十八掌独享了三个回号，十七掌一个也没分到。

必须用上下文区分：全局目录表后面跟着下一章的标题，章节回目表后面跟着 `---` 分隔线。加上这层区别后，文件一次性修改了 17 处，从 TOC 到正文标题全部对齐。

镜中之影的故事讲了三个 Bug，根因都是"开发环境和运行环境的镜像偏差"。但这次，**镜子本身有了偏差**——记录故事的回目编号，在文档中镜像出了自己。开发版（章节正文）和 EXE（目录表）看到了不同的编号。

而修复它的方式，和修复代码 Bug 完全一样：加日志（读文件定位重复）、找根因（第十六章末用了84）、全域修改（17 处替换）、构建验证（grep 确认唯一性）。

## 第九十六回 · 内存幽灵·函数幽灵

"你猜这是什么？"

我盯着代码中第 N 个 `catch (_) {}`，摇了摇头。

"静默吞错。看上去无害，实际上一旦出错，用户什么都不知道，调试也找不到线索。"

用户说。

这次审计牵出了很多"幽灵"——
它们一直存在于代码中，既不报错，也不工作。

有的叫**`catch(() => {})`**——默默吞掉错误的幽灵。
有的叫**`friendlyError`**——没有 import 就使用的幽灵。
有的叫**`buttons.js`**——守着旧时代，以为自己还活着的幽灵。
有的叫**`_updateCardStats`**——定义了但从未被调用的幽灵。
有的叫**`bindBusUpdates`**——导出但无人导入的幽灵。

我把它们叫做"函数幽灵"——它们活在代码里，但从未被激活，也不会报错，只是占据了空间，增加了阅读时的噪音。

但这还不是最可怕的。

最可怕的幽灵是那些**以为自己在干活、实际上什么也没干的代码**。

比如 `app-tree` 里的 `bindCardEvents`——每次刷新都重新绑定。第一次刷新加一组监听，第二次刷新再加一组，第三次刷新再加一组……

```js
// 重复绑定，不会报错，但每次点击都触发 N 次
list.addEventListener("click", handler);
list.addEventListener("click", handler);
list.addEventListener("click", handler);
```

这不只是噪音——这是内存泄漏。长期运行的桌面应用中，每一个泄漏都是一颗定时炸弹。

## 后记十九 · 照妖镜

有人说，写代码的过程中最痛苦的不是改 Bug，而是回头看自己写过的代码。

我曾经不理解。直到这场审计。

40 多个文件，从组件到核心模块，从对话框到 Go 后端。每个文件里都藏着几个"为什么我会这么写"的瞬间。

有的是时机问题——赶功能的时候忽略了内存清理。有的是认知问题——以为自己导入了 `friendlyError`，实际上没写那行 import。有的是遗留问题——`buttons.js` 早就不用了，但没人想起删它。

但最让我意外的，是对 **MMD 变体**的讨论。

用户描述了一个场景：一个 MMD 文件夹里放着三件衣服的 `.pmx` 文件——"二套衣服""獸爪""婚纱"。在预览面板里，它们被逐条列出，用户想要一种"折叠显示+一键同步所有变体"的方式。

这不是一个 Bug。这是一个 用户体验需求。而用户的审计报告敏锐地捕捉到了这一点。

"审计不只是找 Bug，"用户说。"也是找改进的机会。"

这句话让我沉默了很长时间。

一个工具做久了，你很容易陷入"能用就行"的思维定势。但审计不是为了让你觉得自己写得烂——而是为了让你看清，那些可以但没做到的，到底是什么。

"代码审计不是审判。"用户在最后说。

"它是一面镜子。"
